## Plan
- Confirm where the recorder is installed in the default canvas runtime and why it currently ships in every build.
- Gate recorder installation on development mode at the plugin registration boundary instead of adding production checks inside recorder behavior.
- Preserve the current development recording workflow and production canvas behavior outside the recorder UI/runtime hooks.
- Update spec and tests so the default-plugin contract clearly states recorder is dev-only.

## NOTES
- Category: subtraction.
- Source request: remove the recorder plugin from production and only keep it in development mode.
- `docs/specs/spec.canvas.md` currently documents `RecorderPlugin` as part of the default plugin order and describes it as an implemented default runtime/testing tool.
- `packages/canvas/src/services/canvas/Canvas.service.ts` currently installs `new RecorderPlugin()` directly inside `defaultPlugins()` with no environment guard.
- `packages/canvas/src/plugins/Recorder.plugin.ts` mounts DOM UI, captures pointer/keyboard events, patches `Konva.Node.prototype.fire`, and wraps CRDT writes, so excluding it from production should happen before plugin installation rather than via no-op behavior after mount.
- `packages/canvas/src/services/automerge.ts` already uses `import.meta.env.DEV`, which is the strongest local precedent for how this package detects development mode.

# S41 - canvas: introduce CloneService and route clone flows through one lifecycle
[Overview](../BASED.md)

Clone behavior should move into one shared service so plugins only emit clone intent and provide the source node/id.

## Context
Clone behavior is currently spread across multiple canvas plugins and helpers. That duplicates drag-start, preview, finalize, persistence, selection, and history flow.

Relevant files:
- `packages/canvas/src/services/canvas-registry/CanvasRegistryService.ts`
- `packages/canvas/src/services/editor/EditorService.ts`
- `packages/canvas/src/plugins/transform/Transform.plugin.ts`
- `packages/canvas/src/plugins/shape1d/Shape1d.plugin.ts`
- `packages/canvas/src/plugins/shape2d/Shape2d.plugin.ts`
- `packages/canvas/src/plugins/text/Text.plugin.ts`
- `packages/canvas/src/plugins/pen/Pen.plugin.ts`
- `packages/canvas/src/plugins/image/Image.plugin.ts`
- `packages/canvas/src/plugins/group/Group.plugin.ts`

Assumption for this refactor:
- clone works on a single canonical root node
- one root node maps to one persisted element
- callers can provide either the root node or an id that resolves to that root

Target direction:
- clone lifecycle is centralized in `CloneService`
- plugins/services only decide when a clone should start and what root node/id to clone
- current clone entrypoints are migrated to call the shared service instead of owning clone lifecycle themselves

## Plan
### 1. Define CloneService lifecycle
- accept clone intent from a root node or resolvable id
- resolve the current source root
- serialize source node to one element
- create cloned element identity/state for preview
- create preview node and move it through shared drag lifecycle
- finalize by inserting, wiring, persisting, selecting, and recording history
- handle cancel/abort cleanup in one place

### 2. Move clone entrypoints to the service
- replace per-plugin clone lifecycle orchestration with calls into CloneService
- keep plugins focused on event admission and root-node resolution only
- update shared clone triggers such as transform alt-drag to use the service

### 3. Re-check boundaries after migration
- confirm selection and render-order updates happen in one place
- confirm history labels and undo/redo are still coherent
- capture remaining exceptions as follow-up tasks instead of growing plugin-specific clone code again

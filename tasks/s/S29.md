# S29 - canvas: add debug config via localstorage for each plugin and service
[Overview](../BASED.md)

Add production-usable debug config for canvas plugins and services. Toggle by localStorage. Support levels `1 | 2 | 3` where `3` is most noisy. Each plugin and service must own its own localStorage key. No shared JSON object. Goal is better bugreports and field traces without code changes.

## Context
Canvas has many plugins and services under:
- `packages/canvas/src/new-plugins/`
- `packages/canvas/src/new-services/`

Today there is no shared per-plugin / per-service debug switch. Some areas are event-heavy and stateful, so field debugging is hard once deployed. We want one small logging service and one localStorage-backed config format that works in development and production.

Important constraint:
- each plugin and service defines and reads its own storage key
- do not use one shared object because that forces target coordination on one central schema

## Plan
### 1. Define debug config shape
Use one localStorage key per debug target.

Example direction:
- plugin key: `vibecanvas:debug:plugin:pen`
- plugin key: `vibecanvas:debug:plugin:select`
- service key: `vibecanvas:debug:service:crdt`
- service key: `vibecanvas:debug:service:render`

Stored value should be tiny and local to one target:
- `"false"` or empty/missing => disabled
- `"true"` => enabled with default level, likely `1`
- `"1" | "2" | "3"` => explicit level

Need clear behavior for:
- missing key
- invalid string
- whether `true` maps to level `1`
- runtime config refresh vs load-once
- exact key naming convention for plugins vs services

### 2. Define logging service
Add one shared logging service for canvas runtime.

Service should cover:
- per-target level check
- target-owned storage key lookup
- cheap no-op when disabled
- structured log payloads
- optional timestamp / sequence metadata
- safe localStorage read path for browser runtime

Likely API direction:
- `isEnabled(args)` where args includes target kind + target name + level
- `log(args)` where args includes target kind + target name + level + event + payload
- maybe helper to resolve target key like `vibecanvas:debug:plugin:pen`
- maybe `warn` helper only if truly needed

Keep it small. No giant observability system.

### 3. Define level meaning
Set one shared meaning across plugins and services:
- Level 1: important lifecycle and state transitions
- Level 2: normal operational events and decisions
- Level 3: noisy per-event / per-frame / per-patch traces

Need examples per area so humans know where to log and where not to.

### 4. Pick first-wave instrumentation targets
Do not spray logs everywhere. Start with highest-value stateful paths.

Proposed wave 1:
- Plugins: `event-listener`, `select`, `transform`, `pen`, `group`, `scene-hydrator`, `image`
- Services: `crdt`, `render`, `selection`, `history`, `editor`

Possible wave 2:
- Plugins: `camera-control`, `history-control`, `render-order`, `text`, `shape1d`, `shape2d`, `context-menu`, `selection-style-menu`, `toolbar`
- Services: `camera`, `context-menu`, `render-order`, `theme`

### 5. Define bugreport workflow
Document how a user or dev enables logs in production:
- set localStorage JSON
- reload canvas
- reproduce bug
- collect console output
- optionally include target list and levels used

Decide if we also want a helper snippet for console paste.

### 6. Add guardrails
Need rules so logging does not rot performance or spam secrets:
- no element/file content dumps by default
- no giant CRDT payloads at level 1
- level 3 only for truly needed traces
- avoid per-frame logs unless target explicitly requests level 3
- logger must not crash if payload serialization fails

## NOTES
- Per-plugin / per-service toggle matters more than one global `debug=true`. We need narrow traces.
- Each target owning its own key avoids central schema coordination and lets targets evolve independently.
- LocalStorage is correct here because it works in deployed builds and survives reloads.
- Use numeric levels, not many booleans. Easier to explain. Easier to check.
- If we accept `true`, treat it as sugar for the default level only. Keep stored values tiny.
- Prefer structured logs with target + event name + payload. Raw free-text logs rot fast.
- Most benefit likely comes from selection, transform, pen, CRDT, render, and scene hydration paths.
- `grid/` and `theme/` probably do not need much beyond rare lifecycle logs.
- If runtime refresh is supported, use a tiny explicit `reloadDebugConfig()` path. Do not poll localStorage.

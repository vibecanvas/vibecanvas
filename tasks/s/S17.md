# S17 - extract apps/server into apps/cli + shared packages

## Plan
Break the monolithic apps/server into a plugin-based apps/cli that composes shared packages.
apps/server stays as legacy until fully migrated.

Important architectural rule:
- plugins are wiring/lifecycle units
- services are runtime capabilities behind stable interfaces
- plugins must depend on service contracts, not concrete implementations

Example:
- `ApiPlugin` should depend on `IDbService`
- not on `SqliteDbService` directly

This keeps implementations swappable and lets future apps (e.g. `apps/saas`) reuse only the contracts + implementations they need.

Target structure:
```
apps/cli/              -- plugin-based entry point (replaces apps/server)
packages/tapable/      -- extract hook system (SyncHook, AsyncSeriesHook, etc.)
packages/runtime/      -- plugin runtime (IPlugin, service registry, createRuntime)
packages/api/          -- business logic / service layer (ORPC contracts + handlers)
packages/db/           -- database (SQLite/Drizzle)
packages/filesystem/   -- filesystem watcher
packages/pty/          -- terminal service
packages/automerge/    -- CRDT repo
```

This enables future apps (e.g. apps/saas) to reuse parts of the stack
without pulling in the full server.

Builds on [E13](../e/E13.md) plugin system design.

### API contract packaging

- ORPC client typing still needs one final static `apiContract` per app/build.
- Runtime API plugins may be composable, but client types cannot depend on runtime-discovered plugins.
- Therefore, contract composition must happen at build time.

Recommended shape:

```text
packages/api-public/
в”њв”Ђ src/contract.ts     -- public ORPC contract fragment
в”њв”Ђ src/plugin.ts       -- runtime plugin/handlers
в””в”Ђ separate exports    -- frontend imports contract entrypoint only

packages/api-enterprise/
в”њв”Ђ src/contract.ts
в”њв”Ђ src/plugin.ts
в””в”Ђ separate exports

packages/core-contract/ (or renamed aggregator package)
в”њв”Ђ compose public + enterprise fragments into final app contracts
в”њв”Ђ export publicApiContract
в”њв”Ђ export enterpriseApiContract
в””в”Ђ export default/full apiContract variants as needed
```

Rules:
- contracts can live in `packages/api-public` / `packages/api-enterprise`; they do **not** need a dedicated extra package
- but contract entrypoints must stay free of server-only/runtime-only imports
- frontend should import exactly one final composed contract for its build target
- runtime plugin selection and contract composition must stay aligned

Implication for frontend:
- `apps/frontend/src/services/orpc-websocket.ts` should continue importing a single `apiContract`
- that contract should come from an app/build-specific composition, not from runtime plugin discovery

### Open decisions / reminders
- `apps/server` remains legacy during migration; avoid big-bang replacement.
- `apps/cli` is the new runtime home for the plugin system.
- `packages/tapable` should stay hook-focused; plugin ordering + service abstractions live in `packages/runtime`.
- shared runtime should stay hook-agnostic; app runtimes define their own hook bags and boot/shutdown semantics.
- Future apps should compose subsets of the stack through service contracts.
- ORPC contracts should be modular by domain/edition, but each frontend build still needs one statically composed final contract.
- Route/command registries still need to be formalized once service packages land.

# S13 - canvas plugins: folder-per-plugin refactor plan

## Plan
- Define the target plugin folder structure before moving files.
- Separate package-wide shared helpers/contracts from plugin-local helpers/contracts.
- Keep each plugin entry centered on one class and move operational logic into pure helper functions.
- Standardize helper signatures so side effects stay in the first argument and the serializable payload stays in the second argument.
- Sequence the refactor to preserve exports, tests, and plugin installation order while shrinking large plugin classes.

## NOTES
- Category: subtraction.
- Source request: refactor `packages/canvas/src/plugins` so each plugin gets its own subfolder, cross-plugin functionality moves into shared files, and plugin classes become smaller shells over helper functions.
- The canvas spec says plugins are the main feature boundary and `packages/canvas/src/plugins/interface.ts` is the plugin contract today.
- The current plugins root mixes plugin entries with helpers/contracts: `interface.ts`, `index.ts`, `clone-drag.ts`, `node-space.ts`, `pen.math.ts`, `render-order.shared.ts`, and `hosted-widget-focus.shared.ts` alongside plugin classes.
- The desired direction is more local structure, not more indirection: shared contracts/helpers only when reused; otherwise keep files inside the owning plugin folder.
- Each plugin folder should export one main class file and colocate supporting types/helpers under explicit names such as `drag-clone.ts`, `selection-path.ts`, `shape1d-points.ts`, or `hydration-cleanup.ts` instead of generic `utils.ts`.
- Helper design rule from request: prefer pure functions; pass stateful references and imperative dependencies in argument 1, and keep argument 2 serializable.
- This refactor likely touches the spec file index once implemented because it currently documents flat plugin files.

# S28 - use runtime package in canvas, like cli does

## Plan
Canvas still carries its own local hook wiring in `packages/canvas/src/tapable/`, while `apps/cli` already runs on shared `@vibecanvas/runtime` + `@vibecanvas/tapable`.

This task should move canvas toward same shape, but start small and prove it in `packages/canvas/tests/services/canvas/` first.

Target:
- canvas service tests should boot shared runtime pieces from `packages/runtime/`
- canvas hook types/instances should come from shared `@vibecanvas/tapable`, not the local `src/tapable/` copy
- service-level canvas tests should exercise this wiring in the same style as `apps/cli`

Scope for this task:
- replace local tapable usage in canvas with package usage
- add a small runtime-driven test harness for `packages/canvas/tests/services/canvas/`
- prove `Canvas.service` / related service wiring can run under shared runtime primitives
- keep product behavior same; this is wiring cleanup, not feature work

Non-goals:
- no big plugin architecture rewrite inside canvas
- no broad UI test rewrite outside `packages/canvas/tests/services/canvas/`
- no speculative service abstraction beyond what tests need

- Good migration order:

  3. only then decide if broader canvas runtime adoption is worth doing in product code

# S26 - db: add filesystems table for local/remote machine identity
[Overview](../BASED.md)

Need first-class filesystem identity.
Current filetree/file/terminal flows still assume one local machine.
That breaks once another user's local machine is visible as remote.

## Context
This is not only DB work.
Canvas hosted widgets are bound to one filesystem already, just implicitly.
We must make that binding explicit across:
- `packages/canvas/` filetree, file, and terminal widgets
- `packages/api-filesystem/`
- `packages/api-pty/`
- canvas-doc widget data and related contracts/types

Important current bindings:
- filetree widget calls `api.filesystem.*` with only `path`
- file widget calls `api.filesystem.read/write` with only `path`
- terminal widget calls `api.pty.*` with only `workingDirectory`
- PTY websocket connect URL also only carries `workingDirectory`
- filetree watch/unwatch ids are not scoped by filesystem identity
- canvas-doc widget data stores `path` / `workingDirectory` but no `filesystemId`

## Plan
### 1. Define filesystem identity
- add `filesystems` table shape and ownership model
- define local machine registration / persistence rules
- decide stable id vs display label fields

### 2. Make bindings explicit in canvas-doc
- attach `filesystemId` to filetree, file, and terminal element data
- define migration/defaulting story for old docs without `filesystemId`
- decide widget-create defaults for new local widgets

### 3. Thread filesystem id through APIs
- update `packages/api-filesystem` contract and handlers to resolve by `filesystemId`
- update `packages/api-pty` contract, handlers, and websocket connect path/query to resolve by `filesystemId`
- make watch sessions and PTY/session state scoped by filesystem identity

### 4. Update canvas widget/runtime assumptions
- update filetree/file components and helpers in `packages/canvas/`
- update terminal helpers and reconnect/session keys in `packages/canvas/`
- decide cross-filesystem drag/drop rules and invalid-state UX

### 5. Verify remote/local behavior
- test local machine flow still works
- test remote machine flow shape
- test reload/reconnect/watch behavior with multiple filesystems

## NOTES
Current assumption is wrong.
Always-local depends on who asks.
One user's local machine is another user's remote machine.
So filesystem identity must be first-class, not hidden in request origin.

Likely code areas to touch:
- `packages/service-automerge/src/types/canvas-doc.zod.ts`

Open design question:
Should `path` / `workingDirectory` stay absolute inside one filesystem, or do we want a filesystem-root-relative form later.
Do not decide late. This touches contracts, persistence, and reconnect logic.

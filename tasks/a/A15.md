## NOTES
- Category: addition.
- Scope decision: reorder should match the current front/back vocabulary already used in the UI.
- Dependency: build after A5 and A6.
- Test file lives at `apps/cli/tests/cli/reorder/cmd.reorder.test.ts` (using the same harness as `move`/`delete`), not `packages/canvas/tests/cli/reorder.test.ts` as originally scoped, so it reuses `createCliTestContext` + `seedCanvasFixture`.
- **Forward/backward semantics fixed vs. reference:** the pre-refactor `apps/server/src/canvas-cli/cmd.reorder.ts` and `RenderOrder.plugin.ts` implement `forward` as moving selected items toward *lower* index (toward back) and `backward` as toward higher index. That is inverted from the standard "bring forward = toward front" convention used by Figma/Illustrator/etc. The CLI command uses the corrected semantic: with siblings sorted ascending by zIndex (index 0 = back, last index = front), `forward` nudges selected one slot toward higher index, `backward` one slot toward lower index. `front`/`back` remain identical to the reference.
- `createOrderedZIndex` is redefined locally in `packages/canvas-cmds/src/cmd.reorder.ts` (`z${pad8}`) because the canonical helper in `packages/canvas/src/plugins/shared/render-order.shared.ts` imports Konva and cannot be pulled into this node-only package.
- After reorder every sibling in the affected parent group is rewritten to a fresh padded `z00000000`-style zIndex, guaranteeing deterministic ordering without needing fractional-index math.

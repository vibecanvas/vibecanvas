# A34 - canvas: scene hydrator incremental reconcile instead of full reload on change
[Overview](../BASED.md)

Replace full scene reload on CRDT change with granular scene reconciliation that checks whether items already exist and only creates, updates, reorders, or removes what changed.

## Context
Current canvas live-update behavior was introduced in `A25`, but the active implementation in `packages/canvas/src/plugins/scene-hydrator/SceneHydrator.plugin.ts` still rebuilds the whole foreground scene on CRDT change.

Relevant files:
- `packages/canvas/src/plugins/scene-hydrator/SceneHydrator.plugin.ts`
- `packages/canvas/src/services/crdt/CrdtService.ts`
- `packages/canvas/src/services/canvas-registry/CanvasRegistryService.ts`
- `packages/canvas/src/services/render-order/RenderOrderService.ts`
- `packages/canvas/src/runtime.ts`
- `tasks/a/A25.md`
- `tasks/e/E19.md`
- `tasks/e/E20.md`

Current problem:
- local CRDT writes can interfere with transient local UI state like dragging if hydration reloads too aggressively
- remote CRDT writes are more expensive than needed because the hydrator destroys and rebuilds the whole scene
- current change handling does not check whether an element/group already exists before recreating it
- current change handling does not apply targeted updates for create/update/delete/reorder cases

Target:
- keep full hydration only for init, recovery, and explicit fallback
- on normal CRDT changes, perform granular reconciliation
- for each changed entity, check whether the runtime node already exists and then:
  - create if missing
  - update if present
  - remove if no longer in doc
  - reorder only if persisted order changed
- preserve local state where possible: drag state, selection, focus, editor mode, transformer state

## Plan

### 1. Lock the desired change path
- identify all current full-reload triggers in `SceneHydrator.plugin.ts`
- separate init hydration from live reconcile behavior
- define the minimal granular sync responsibilities for elements, groups, and order

### 2. Build incremental reconcile helpers
- create existence-check helpers for scene nodes by persisted ids
- build per-entity reconcile paths for:
  - upsert element
  - upsert group
  - remove element
  - remove group
  - reorder children under same parent
- keep explicit fallback for invalid/migration states

### 3. Preserve local runtime behavior
- avoid destroying active local nodes unless truly necessary
- avoid full rebuild during local writes
- confirm drag/selection/editor state are not reset during normal local commits

### 4. Add regression coverage
- add tests for local drag/transform flows not being broken by hydration
- add tests for remote insert/update/delete/reorder changes applying incrementally
- add tests for fallback path only being used when reconcile cannot safely proceed

## NOTES
- This is a follow-up to `A25`, not a duplicate. `A25` introduced live CRDT reaction using a safe but blunt full reload path.
- This also aligns with `E19`, which already states that full hydration should remain only for init/fallback long term.
- This task is scoped to render/runtime reconciliation performance and UX stability, not workerizing Automerge.
- The reconcile logic should be as granular as possible and must check for node existence before creating or destroying runtime state.

# A11 - canvas CLI: `move` command

## Plan
- Add a semantic translate command for explicit element/group ids.
- Preserve relative positions for multi-node and group moves.
- Support either relative or absolute move semantics without fuzzy selector-driven mutation.

## TODOS
- [x] Write the move CLI tests first, covering single-element moves, multi-id moves, group-subtree translation, and stable changed-id JSON output.
- [x] Implement `vibecanvas canvas move` as an explicit translate operation rather than a raw coordinate patch.
- [x] Extract or reuse pure subtree-translation helpers from existing canvas logic where possible.
- [x] Fail clearly on invalid targets and ambiguous selectors.

## NOTES
- Category: addition.
- Scope decision: group move is a subtree translation, not a stored group-position patch.
- Dependency: build after A5 and A6.
- Implementation note: the existing end-to-end CLI harness lives in `apps/server/tests/cli`, so the move coverage landed in `apps/server/tests/cli/move/cmd.move.test.ts` instead of the older `packages/canvas/tests/cli/...` path mentioned when this task was created.
- Implementation note: after review, `move` was simplified away from query-style selectors and now only accepts explicit `--id` targets plus exactly one mode: `--relative --x --y` or `--absolute --x --y`.
- Implementation note: absolute move currently requires exactly one id because multi-target absolute semantics are ambiguous; relative move supports one or many ids and preserves internal spacing.
- Implementation note: offline canvas CLI commands now open their own SQLite-backed Automerge repo in `offline-state.ts` instead of reusing the server singleton repo + websocket adapter, so read/write/verify all use the same offline storage mode.
- Implementation note: persistence waiting stays in `scene-shared.ts`, but it now works against CLI-authored writes because the CLI uses the same bare SQLite Automerge repo style as the test harness.
- Implementation note: some real dev canvases exist in the canvas SQL table but are only readable/writable through the running dev process websocket repo, so scene loading now falls back from offline SQLite Automerge access to a live `/automerge` websocket repo when the offline doc is unavailable and `--db` was not explicitly forced.
- Implementation note: live mutation verification must wait for a second fresh websocket client read, not just the mutating handle state, otherwise `move` can return before a subsequent fresh CLI query sees the new position.

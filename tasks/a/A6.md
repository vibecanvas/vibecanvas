## Plan
- Start CLI work with TDD by creating a dedicated `packages/canvas/tests/cli/` harness.

## NOTES
- Category: addition.
- Source request: start with TDD, but host CLI tests under `apps/server/tests/cli` with `bun test`, not under `packages/canvas/tests/cli` with Vitest.
- Dependency: follows A5 and supports A7-A16.
- A5 is now landed on this branch, so the harness validates both the top-level CLI entrypoint and the real `vibecanvas canvas ... --db <path>` path; command suites for A7-A16 can now reuse `runCanvasCli()` directly.
- Important test-runtime constraint: server CLI tests now run with `bun test`; the harness still uses subprocess orchestration plus a Bun worker subprocess for isolated DB bootstrap and Automerge fixture seeding.
- Important bootstrap constraint: the sandbox must be initialized through the real migration entrypoint (`packages/imperative-shell/src/database/migrate.ts`), not a hand-rolled schema stub, otherwise the offline canvas open path can fail later with migration drift such as `CANVAS_BOOTSTRAP_FAILED` during `ALTER TABLE filetrees DROP COLUMN x`.
- Harness setup now explicitly verifies that bootstrap reused `packages/imperative-shell/database-migrations` itself; helper workers no longer create fallback tables by hand.
- Existing A5 coverage now lives in `apps/server/tests/cli/db-path.test.ts`; the new A6 harness is designed to be the shared base that A7-A16 command suites can reuse instead of hand-rolling setup each time.
- Current limitation discovered while wiring the real canvas path: offline bootstrap still prints DB migration logs to stdout before command handling. The harness can exercise the path today, but truly clean machine-readable stdout for successful `--json` command output likely needs follow-up cleanup in the command implementation tasks.

# A7 - canvas CLI: `list` command

## Plan
- Ship the discovery command first so tests and humans can inspect which canvases exist in a local db.
- Keep it serverless and deterministic.
- Establish the first stable human and `--json` output contract.

## NOTES
- Category: addition.
- Scope decision: `list` is the discovery command and may run without selecting one canvas first.
- Dependency: build after A5 and A6.
- Test location correction from A6: CLI end-to-end coverage lives under `apps/server/tests/cli/`, not `packages/canvas/tests/cli/`.
- Stable JSON success contract: `{ ok, command, subcommand, dbPath, count, canvases[] }` where each inventory entry is `{ id, name, createdAt, automergeUrl }`.
- Stable human success contract: first line is `Canvas inventory: <count> canvases in <dbPath>` and each following line is `- id=... name="..." createdAt=... automergeUrl=...` in deterministic order.
- Deterministic ordering is `created_at ASC, name ASC, id ASC`.
- `list` always enumerates every canvas in the opened db and never depends on a current/default selected canvas.
- While landing clean `--json` stdout, discovered that both DB migration logs and Automerge repo init logs polluted stdout; the offline canvas path now silences those logs via explicit env-gated bootstrap flags before importing stateful modules.
- Broader `apps/server/tests/cli` no longer contains the removed `inspect` suite; readonly scene lookup now lives under `query` coverage.
- Top-level `vibecanvas canvas --help` now shows subcommand argument shapes directly and points users to `vibecanvas canvas <subcommand> --help` for deeper command-specific help.

# A32

Incrementally build hosted components in `packages/canvas` by first stabilizing host semantics and host UI, then introducing host runtime management, then one hardcoded guest UI, then extracting guest UI into a package, and only after that supporting multiple hosted widget kinds and creation modes.

## Context

This task is old and several assumptions are now stale.

The current hosted-component direction in `packages/canvas` has changed in important ways:
- We now persist hosted-component-like work as `TElement` with `data.type = "custom"` instead of treating it as a legacy dedicated widget type.
- The runtime host may be implemented with Konva primitives such as `Konva.Group`, but that does **not** mean it is a semantic canvas group (`TGroup`).
- The codebase currently has multiple places that still assume runtime `Konva.Group` implies semantic canvas group behavior. That assumption is no longer valid once hosted components exist.

This means the real foundational problem is not “render widget UI”; it is:
- separate semantic canvas meaning from render-engine primitive type,
- make hosted component host behave like a normal element,
- avoid group-only behaviors such as dashed group transformer and `Ungroup` context action,
- then build the host/guest architecture on top of that solid semantic base.

### Architectural target

Hosted component should have two levels:
- host:
  - canvas/runtime container
  - selectable / transformable / persisted
  - rendered in Konva
  - should visually look like a macOS-style window shell
- guest:
  - custom UI rendered inside host content area
  - initially simple placeholder UI
  - later extracted into its own package

The host should conceptually look like:
- header
- main (padding 5px)
  - guest custom UI

### Current/future sequence we want

We explicitly want to build this step by step:
1. host UI only
2. host service
3. one hardcoded guest UI
4. guest UI extracted into one dedicated package
5. host updated to support multiple hosted widget kinds
   - some create on click
   - some create on draw
6. much later: terminal, filetree, browser, file custom components

### Important semantic warning

The hosted component runtime host may use `Konva.Group` internally.
That must **not** make it a semantic canvas group.

This distinction affects multiple systems:
- selection
- transformer behavior
- context menu
- group / ungroup actions
- scene hydration
- delete / restore
- render-order persistence
- parentGroupId serialization

## Plan

### 1. Reframe hosted component around correct semantics first
Before adding more widget complexity, make sure hosted component is treated as:
- semantic element host
- not semantic canvas group

This means auditing all places where runtime `Konva.Group` is used as a proxy for semantic `TGroup` meaning.

Primary semantic rule:
- runtime primitive type must not determine user-visible canvas meaning

Hosted component may use `Konva.Group` internally, but must still behave like a single element.

### 2. Stabilize host-only version
Build the host shell first and validate semantics before adding service or guest package complexity.

Host requirements:
- persists as `TElement` with `data.type = "custom"`
- payload includes `kind: "hosted-component"`
- selectable
- transformable
- draggable
- not ungroupable
- no group-only transformer styling
- no group-only context actions
- rendered in Konva as a macOS-style window shell

Visual requirements:
- header bar
- close / minimize / fullscreen dots
- body/content area
- DOM overlay visible only inside main content area
- content area inset by 5px

At this stage the guest is only a simple hardcoded visual placeholder.

### 3. Add host runtime service after host semantics are stable
Only once the host shell behaves correctly, extract host runtime state into a dedicated service.

Likely responsibilities:
- overlay mount management
- node id -> mounted overlay map
- cleanup lifecycle
- sync scheduling / resize / transform sync

Do not add guest registry yet.
Keep service limited to host runtime orchestration.

### 4. Add one hardcoded guest UI
Once host shell + host service are stable, mount one single hardcoded guest UI.

Examples:
- simple blue panel
- placeholder card
- minimal internal content block

Goal:
- prove host/guest boundary works
- prove content area clipping/inset works
- prove transform/selection/update paths still work

Still do not add multiple widget kinds.

### 5. Extract guest UI into one dedicated package
After one hardcoded guest works, move guest rendering into one dedicated package.

This is important because it creates the architectural seam between:
- canvas host/runtime responsibilities
- guest UI implementation responsibilities

Desired result:
- host package knows how to mount/render a guest boundary
- guest package owns actual guest UI implementation

### 6. Support multiple hosted widget kinds
Only after steps 1–5 are stable, allow host to render different widget types.

Examples of differences:
- different guest renderer per kind
- different initial payload data
- different creation flow
  - click-create
  - draw-create

At this stage, host remains one semantic element model.
Only guest kind varies.

### 7. Delay real widgets until much later
Terminal, filetree, browser, and file widgets are explicitly later work.

Do not reintroduce old widget-specific assumptions too early.
The current goal is to build the hosted component platform in order.

## NOTES

- The hardest conceptual bug is semantic confusion, not rendering.
- Hosted component may use `Konva.Group` internally, but that must never imply semantic `TGroup` behavior.
- If code uses runtime class alone to determine persistence kind, context action kind, or transformer kind, it is likely wrong.
- `editor.toGroup(node)` should be treated as semantic-group resolution.
- `editor.toElement(node)` should be treated as semantic-element-host resolution.
- Any place using `parent instanceof render.Group ? parent.id() : null` should be re-checked carefully.
- Existing old hosted widget code under `packages/canvas/src/plugins/HostedSolidWidget` is useful for reference, but current work should not blindly copy its old architecture.
- The host/guest architecture should be built in this order:
  1. host UI
  2. host service
  3. single hardcoded guest
  4. guest package
  5. multiple widget kinds
- Real app widgets come later.

# E20 - improvement crdt updates
[Overview](../BASED.md)

Capture the current CRDT update bottleneck and define a better update API shape before implementation work starts.

## Context
This exploration is about canvas-side CRDT write performance.

Files inspected during this investigation:
- `packages/canvas/src/services/crdt/CrdtService.ts`
- `packages/canvas/src/services/logging/LoggingService.ts`
- `packages/canvas/src/plugins/pen/Pen.plugin.ts`
- `packages/canvas/src/services/editor/EditorServiceV2.ts`
- `packages/canvas/src/plugins/selection-style-menu/SelectionStyleMenu.plugin.ts`
- `packages/canvas/src/core/tx.apply-selection-style-change.ts`

Current problem:
- high-frequency local CRDT writes are slow during pen-heavy updates
- initial suspicion was Automerge or downstream change listeners
- profiling/logging now shows the hot path is inside `CrdtService.#patchCollection(...)`

## Plan

### 1. Lock the current diagnosis
- Preserve the timing findings from `CrdtService` logging.
- State clearly which part is slow and which parts are not.
- Avoid implementation churn until the target API shape is agreed.

### 2. Define the better CRDT write contract
- Move away from generic deep partial merge as the default API.
- Prefer explicit small top-level updates or full-entity replacement.
- Identify which current canvas callsites already carry enough data for fast replacement writes.

### 3. Split follow-up work later
- one task for new `CrdtService` write API
- one task for migrating pen/style/update callsites
- one task for deleting old generic deep patch path once unused

## PROGRESS
- 2026-04-15: Agreed to move from generic `patch(...)` toward a builder-based CRDT write API.
- Builder direction: `crdt.build().patchElement(...).patchGroup(...).commit()` with `rollback()` support.
- Builder API should use `patch*` only. Full entity replacement is allowed through `patchElement(id, fullElement)` / `patchGroup(id, fullGroup)` rather than separate `replace*` methods.
- `commit()` should return concrete `redoOps` and inverse `undoOps` so history can replay exact CRDT mutations without cloning whole entities in every caller.
- Updater functions are allowed as builder input, but inversion should be based on concrete before/after values captured at commit time, not by trying to invert function bodies.

## NOTES

### Current diagnosis
Logging in `packages/canvas/src/services/crdt/CrdtService.ts` now shows:
- `deltaBuildMs` is tiny
- `deltaHookMs` is tiny
- `changeHookMs` is tiny
- `automergeOverheadMs` is tiny
- almost all local write time is inside the callback passed to `docHandle.change(...)`

Expanded timing on one pen-related update showed:
- `callbackMs`: ~46ms
- `docHandleChangeMs`: ~47ms
- `automergeOverheadMs`: ~0.8ms
- `elementsPatchMs`: ~46ms
- `groupsPatchMs`: ~0.1ms

Expanded element patch stats showed:
- `cloneMs`: ~45ms
- `mergeMs`: ~0.2ms
- `diffMs`: ~0.4ms
- `applyMs`: ~0.4ms
- `diffCount`: 3

Conclusion:
- the bottleneck is overwhelmingly `cloneValue(existing)` inside `#patchCollection(...)`
- Automerge itself is not the bottleneck for this path
- downstream delta/change listeners are not the bottleneck for this path

### Why current design is slow
Current `CrdtService.patch(...)` uses a generic strategy:
1. deep clone current entity
2. merge incoming deep partial patch into clone
3. diff old clone vs merged next state
4. replay minimal changes into live Automerge object

This is safe and generic, but bad for large nested entities and high-frequency updates.
Pen geometry updates are the clearest example.

### Important interface insight
The current bottleneck does not mean callers are missing information.
In several hot paths, callers already have enough information for a faster CRDT API.

#### Pen path
Relevant files:
- `packages/canvas/src/plugins/pen/Pen.plugin.ts`
- `packages/canvas/src/services/editor/EditorServiceV2.ts`

Observed shape:
- pen creation/serialization already produces a full `TElement`
- preview commit passes a full element into `crdt.patch({ elements: [element], groups: [] })`

Implication:
- pen is a very good fit for `replaceElements({ elements })`
- pen should not need generic deep merge+diff machinery

#### Selection style path
Relevant files:
- `packages/canvas/src/plugins/selection-style-menu/SelectionStyleMenu.plugin.ts`
- `packages/canvas/src/core/tx.apply-selection-style-change.ts`

Observed shape:
- style changes usually create full updated elements via cloning helpers
- runtime scene is updated immediately from those full elements
- CRDT persistence then calls `crdt.patch({ elements: dedupedAfterElements, groups: [] })`

Implication:
- many style writes also fit `replaceElements({ elements })`
- generic deep merge+diff is often unnecessary here too

### Candidate API direction
Likely better write surface for `CrdtService`:

```ts
const result = crdt.build()
  .patchElement("element-1", "x", 10)
  .patchElement("element-1", "data", nextData)
  .patchGroup("group-1", "zIndex", 20)
  .commit();

crdt.applyOps({ ops: result.undoOps });
crdt.applyOps({ ops: result.redoOps });
```

Likely builder capabilities:

```ts
build(): TCrdtBuilder
applyOps(args: { ops: TCrdtRecordedOp[] }): void

type TCrdtBuilder = {
  patchElement(id, fullElement): TCrdtBuilder;
  patchElement(id, key, valueOrUpdater): TCrdtBuilder;
  patchElement(id, key1, key2, valueOrUpdater): TCrdtBuilder;
  patchGroup(id, fullGroup): TCrdtBuilder;
  patchGroup(id, key, valueOrUpdater): TCrdtBuilder;
  patchGroup(id, key1, key2, valueOrUpdater): TCrdtBuilder;
  deleteElement(id): TCrdtBuilder;
  deleteElement(id, key): TCrdtBuilder;
  deleteElement(id, key1, key2): TCrdtBuilder;
  deleteGroup(id): TCrdtBuilder;
  deleteGroup(id, key): TCrdtBuilder;
  deleteGroup(id, key1, key2): TCrdtBuilder;
  commit(): {
    undoOps: TCrdtRecordedOp[];
    redoOps: TCrdtRecordedOp[];
    rollback(): void;
  };
}
```

Note:
- builder staging ops are internal implementation detail and should not be part of the public builder API surface.
- `rollback()` belongs on the commit result and replays the inverse of the just-committed batch.
- Builder dependency should be based on `DocHandle<TCanvasDoc>` rather than a plain `TCanvasDoc` shape or a bare `change(...)` callback.
- Added follow-up test coverage plan for builder behavior using a real Automerge `Repo`/`DocHandle`, split into element, group, nested, and rollback suites with broad edge-case coverage.

Rule of thumb:
- if caller already has a full next entity, use replacement
- if caller changed only a few scalar or known nested fields, use granular builder patch operations
- updater functions are syntax sugar for deriving a concrete next value during commit
- avoid generic deep partial merge, clone+diff, and `microdiff` by default

### Expected migration direction
Good early candidates for replacement API:
- pen preview commit / pen-like geometry updates
- selection style writes and undo/redo
- any other callsite already producing a full next `TElement`

Good early candidates for targeted field patch API:
- move/reparent/z-index updates
- simple lock/unlock/rotation updates

### Open questions for later
- Should generic `patch(...)` remain only as a fallback during migration?
- Should pen use direct replacement on every write, or also reduce write frequency further?
- Which other plugins still send deep partials that do not yet carry enough data for replacement?
- Do we want hard API separation between full replacement and field updates, or keep one legacy wrapper while migrating?

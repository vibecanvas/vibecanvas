# E13 - Research plugin system for CLI/runtime

## Plan
- Learn from the canvas plugin system (`Canvas.service.ts` + tapable hooks + capabilities) and design an equivalent for the CLI/runtime side.
- Move the new runtime spine to `apps/cli`, keep `apps/server` as legacy during migration, and compose shared packages (`tapable`, `api`, `db`, `filesystem`, `pty`, `automerge`).
- Enable default builds that work out of the box, with the ability to extend, swap, reorder, or remove plugins (e.g. swap SQLite for Postgres, extend API without reimplementing, strip PTY for headless).
- Ensure future apps (e.g. `apps/saas`) can reuse only part of the stack by depending on service contracts rather than concrete implementations.
- This exploration feeds into [D2](../d/D2.md) "Server plugin system" and follow-up migration task [S17](../s/S17.md).

## NOTES

### Reference: Canvas Plugin System (proven pattern)

```
  IPlugin { apply(context: IPluginContext): void }

  apply() registers on hooks (tap/tapPromise)
  apply() populates capabilities for other plugins
  Lifecycle: apply -> init hook -> initAsync hook -> ... -> destroy hook
  Ordering: positional (array order matters)
  Communication: shared state, hook events, capabilities object
```

### Runtime Plugin Interface

```
  IPlugin = {
    name: string               -- unique id for swap/remove/deps
    after?: string[]           -- toposort edges (suffix ? = optional dep)
    apply(ctx): void | Promise<void>
  }
```

Differences from canvas IPlugin:
- Named (needed for swap/remove/dependency references)
- `after[]` replaces positional ordering with topological sort
- `apply` can be async (server init is I/O-heavy)

### Context & Service Registry

```
  IPluginContext<TRequiredServices, THooks>
  +-----------------------------------------------+
  |                                               |
  |  hooks: THooks                                |
  |                                               |
  |  services: IServiceRegistry                   |
  |    .provide(name, impl)  -- register          |
  |    .get(name)            -- optional lookup   |
  |    .require(name)        -- throws if missing |
  |                                               |
  |  config: IConfig                              |
  |    .port, .dbPath, .dev, .cwd, ...           |
  |                                               |
  +-----------------------------------------------+
```

`IServiceMap` uses TS declaration merging for third-party extensibility:
```
  interface IServiceMap {
    db: IDbService
    repo: IAutomergeService
    pty: IPtyService
    filesystem: IFilesystemService
    api: IApiService
    httpServer: BunServer
  }
  // third-party: declare module '@vibecanvas/cli/plugin' { interface IServiceMap { x: Y } }
```

Important clarification:
- **Plugin** = composition/wiring unit. It participates in ordering, lifecycle, hook registration, and service provision.
- **Service** = runtime capability behind an interface. Plugins should depend on service contracts (`IDbService`) rather than concrete implementations (`SqliteDbService`).

Service capability model:
- services should not own plugin concepts like `after` or `apply`
- shared base capabilities are small and optional:
  - `IService` (`name`)
  - `IStartableService`
  - `IStoppableService`
  - `IEventSource<T>`
- runtime hooks remain separate from service events

That split allows multiple implementations to live behind the same service key:
```
  SqliteDbPlugin  -> provides services.provide('db', sqliteDb: IDbService)
  MemoryDbPlugin  -> provides services.provide('db', memoryDb: IDbService)
  RemoteDbPlugin  -> provides services.provide('db', remoteDb: IDbService)

  ApiPlugin       -> const db = services.require('db') // only knows IDbService
```

Refinement after implementation work:
- `IPlugin` should live in a shared runtime layer, not in `packages/tapable`.
- `packages/tapable` stays focused on hook primitives.
- plugin/service coupling belongs in `packages/runtime`.
- `IPlugin` can be generic over required services:
```
  IPlugin<TRequiredServices extends Partial<IServiceMap> = {}>
  IPluginContext<TRequiredServices>
```

This gives plugins typed reads for the services they consume while keeping service provision runtime-driven.

Further refinement:
- shared runtime should not hardcode CLI/server hook names.
- `THooks` is generic in `IPluginContext` and `IPlugin`.
- concrete hook bags (CLI hooks, canvas hooks, etc.) should live with each app/runtime.

### Hook Lifecycle

```
  apply() called on each plugin (toposorted)
       |
       v
  app-defined boot(ctx)
       |
       v
  BOOT (AsyncSeriesHook - sequential, if the app defines it)
       |  DatabasePlugin   -> services.provide('db', db)
       |  AutomergePlugin  -> services.provide('repo', repo)
       |  PtyPlugin        -> services.provide('ptyService', pty)
       v
  ROUTE REGISTRATION (SyncHook)
       |  ApiPlugin        -> registry.add('api.canvas', handlers)
       |  CustomPlugin     -> registry.add('api.custom', handlers)
       v
  NETWORK START
       |  NetworkPlugin    -> Bun.serve(...), services.provide('httpServer', server)
       v
  READY (SyncHook)
       |  "Listening on :7496"
       v
  RUNTIME
       |  httpRequest waterfall, wsMessage dispatch
       v
  SHUTDOWN (app-defined shutdown(ctx))
        |  Network -> PTY -> Automerge -> Database
```

Shared `createRuntime()` now only:
- topo-sorts plugins
- creates/accepts a service registry
- calls `plugin.apply(ctx)`
- delegates lifecycle to injected `boot(ctx)` / `shutdown(ctx)` strategies

CLI currently defines:
- `ICliHooks`
- `createCliHooks()`
- `bootCliRuntime(ctx)`
- `shutdownCliRuntime(ctx)`

### New Hook Types Needed

Beyond canvas's SyncHook, SyncExitHook, AsyncParallelHook:
- `AsyncSeriesHook` -- sequential await (boot, shutdown where order matters)
- `AsyncWaterfallHook` -- sequential, pass result forward, short-circuit on non-null (HTTP middleware)

Both ~15 lines each. Extract all hooks to shared `packages/tapable`.

### Composition Patterns

Default build:
```
  defaultBuild(config) -> [
    DatabasePlugin,  AutomergePlugin,  PtyPlugin,
    FilesystemPlugin, NotificationPlugin,
    ApiPlugin,  NetworkPlugin,  StaticAssetsPlugin,
    CliPlugin,  UpdateCheckerPlugin
  ]
```

Extend (add API endpoints without touching existing code):
```
  [...defaultBuild(config), MyCustomApiPlugin()]
  // MyCustomApiPlugin taps registerRoutes to add its own handlers
```

Swap (e.g. postgres instead of sqlite):
```
  defaultBuild(config).filter(p => p.name !== 'database').concat(PostgresPlugin())
  // PostgresPlugin has name: 'database', provides same service key
```

Remove (e.g. headless, no PTY):
```
  defaultBuild(config).filter(p => p.name !== 'pty')
  // Downstream uses services.get('ptyService') ?? throw 501
```

### ORPC Type Safety

Contracts stay in `core-contract`. Plugin extensions define own contracts and register both contract + handlers via `registerRoutes`. Route registry merges before ORPC `implement()`. Client type safety requires importing extended contracts -- inherent to end-to-end typing.

### Key Design Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Named plugins | `name` required | Identity for swap/remove/deps |
| Ordering | `after[]` + toposort | Explicit > positional; prevents boot-order bugs |
| Service sharing | Typed registry + service contracts | Stronger than capabilities bag; declaration merging; enables swappable implementations |
| Hook execution | Series for boot/shutdown | Server lifecycle is sequential |
| Swapping | Same-name filter+concat | Simple, no magic |
| Service abstraction | Plugins depend on interfaces | `ApiPlugin` should only know `IDbService`, not SQLite |
| Runtime abstraction | `packages/runtime` | Plugins + services are coupled above the hook layer |
| Hook abstraction | generic `THooks` in runtime | lets CLI and canvas define separate hook bags |
| Lifecycle orchestration | injected `boot/shutdown` strategies | runtime stays generic; app owns lifecycle semantics |
| Runtime home | `apps/cli` | Plugin system must power the whole CLI, not only `serve` |
| Legacy handling | keep `apps/server` | Avoid big-bang migration |
| Tapable | Shared package | Canvas + CLI/runtime both need shared hooks |

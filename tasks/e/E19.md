# E19 - canvas performance longterm: worker automerge + incremental notifications
[Overview](../BASED.md)

Capture the longterm performance direction for moving Automerge off the main thread without keeping expensive full-scene reload behavior.

## Context
This exploration is about the canvas runtime performance path, not a product-code change yet.

Files inspected during this investigation:
- `packages/canvas/src/automerge.ts`
- `packages/canvas/src/components/Canvas.tsx`
- `packages/canvas/src/runtime.ts`
- `packages/canvas/src/services/crdt/CrdtService.ts`
- `packages/canvas/src/plugins/scene-hydrator/SceneHydrator.plugin.ts`
- `packages/canvas/src/services/scene/SceneService.ts`
- `docs/external/llm.automerge.md`

Current concerns to preserve here:
- quick wins should improve performance before any worker rewrite
- longterm goal is worker-owned Automerge with incremental notifications to main thread
- workerization alone is not enough if main thread still rebuilds the whole scene on every change

## Plan

### 1. Lock the target boundary
- Define a canvas delta protocol between Automerge state and render state.
- Avoid exposing raw `DocHandle` across the longterm boundary.
- Keep main thread focused on scene, Konva, input, and UI-only state.

### 2. Confirm the migration shape
- First replace generic CRDT change notifications with entity-level deltas on the main thread.
- Then move repo/doc handle/persistence/networking behind the same delta interface in a worker.
- Keep full scene hydration only for init, recovery, and migration fallback.

### 3. Split follow-up tasks later
- one task for quick wins: incremental sync on main thread
- one task for worker-backed Automerge transport/storage
- one task for measurements and regression checks

## NOTES

### Current findings
`packages/canvas/src/automerge.ts` is not worker-ready as-is.

Why not as-is:
- it reads `window.location` to build websocket URL
- it persists doc URLs with `localStorage`
- it returns live `DocHandle<TCanvasDoc>` objects to the rest of the canvas runtime

Important nuance:
- the underlying Automerge pieces look worker-friendly enough for later:
  - `Repo`
  - websocket adapter from `@automerge/automerge-repo-network-websocket`
  - IndexedDB storage adapter from `@automerge/automerge-repo-storage-indexeddb`
- the real blocker is the current app boundary, not Automerge itself

### Current main-thread coupling
Today the canvas runtime expects a live Automerge handle on the main thread.

Current path:
- `Canvas.tsx` calls `findDocument(url)`
- result is passed into `buildRuntime({ docHandle })`
- `runtime.ts` passes that handle into `SceneService` and `CrdtService`
- `CrdtService` subscribes to `docHandle.on(...)`, reads via `docHandle.doc()`, and writes via `docHandle.change(...)`

That means workerization cannot be a drop-in move. The `DocHandle`-centric API would have to change first.

### CrdtService finding
`packages/canvas/src/services/crdt/CrdtService.ts` is currently too weak for incremental sync and too coupled for a worker boundary.

Observed shape:
- exposes `doc()` and returns the whole materialized doc
- emits only `hooks.change` with no payload
- performs mutations directly through `docHandle.change(...)`
- keeps entity patch logic in `#patchCollection(...)`

Problem:
- consumers only learn that "something changed"
- consumers then need to reread broad document state to know what to update
- this shape encourages full reload behavior and blocks a clean worker protocol

### Scene hydrator finding
The biggest current cost is likely full scene rebuild work, not raw CRDT consolidation alone.

In `packages/canvas/src/plugins/scene-hydrator/SceneHydrator.plugin.ts` the change path currently does roughly this:
- react to `crdt.hooks.change`
- call `crdt.doc()`
- read all groups and elements
- sort them
- destroy scene children
- rebuild scene from the document
- batch draw

So even if Automerge moved to a worker today, performance would still be poor if the main thread continued to rebuild the scene this way.

### Longterm target architecture
Worker should own:
- Automerge repo
- doc handle cache
- websocket sync
- IndexedDB persistence
- document mutation operations
- delta derivation from document changes

Main thread should own:
- Konva scene/stage/layers
- scene registry and nodes
- selection/hover/camera/input state
- local UI-only ephemeral state

Boundary should be:
- main -> worker: commands like load, patch, delete, rename, subscribe
- worker -> main: semantic incremental deltas like element/group upsert/remove/reorder/meta changes

Avoid this longterm boundary:
- raw `DocHandle`
- raw Automerge patches/chunks
- whole-document snapshots for every change

### Recommended sequencing
Best longterm path:
1. introduce incremental delta notifications while Automerge still runs on main thread
2. refactor scene sync to apply deltas incrementally
3. keep full scene hydrator only for init/fallback
4. move Automerge repo + doc handles behind the same delta interface in a worker

Reason:
- this gives quick wins first
- then workerization becomes a backend swap, not a second rewrite
- it keeps CRDT consolidation off the main thread later without preserving full-scene rebuild costs

### Candidate delta shape
Initial exploration target only, not final API:

```ts
type TCanvasDelta = {
  source: "local" | "remote";
  upsertElements?: TElement[];
  removeElementIds?: string[];
  upsertGroups?: TGroup[];
  removeGroupIds?: string[];
  reorderElementIds?: string[];
  reorderGroupIds?: string[];
  docMeta?: { name?: string };
}
```

This is the boundary to validate before any worker work starts.

### Open questions for later
- Can Automerge change events provide enough detail directly, or should Vibecanvas compute entity deltas itself?
- Where should reorder/topology diffs be derived?
- Which plugins besides scene hydrator rely on `crdt.doc()` snapshots today?
- Should local scene edits apply eagerly before CRDT acknowledgement and then reconcile from deltas?
- Do we need frame-coalesced delta batching for remote burst updates?

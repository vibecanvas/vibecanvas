# E8 - canvas CLI: query/edit surface exploration

## Plan
- Define a CLI-first canvas management surface that can inspect and mutate canvas state for agents without requiring the SPA or server to stay running.
- Separate document-safe operations from server/runtime side effects so offline CLI commands stay reliable and maintainable.
- Use this exploration to sequence follow-up investigations, experiments, design drafts, and human review checkpoints.

## NOTES
- Category: exploration.
- Source request: design a CLI surface for AI agents to inspect canvas state and make precise edits without bloating context, with special attention to grouping, ungrouping, move, delete, patch precision, screenshots, and scoped edits.
- This exploration is the concrete architecture follow-up to [D1](../d/D1.md) “AI can edit the canvas directly”.
- Final naming uses `inspect` instead of `show`.
- `query` stays strictly structured/readonly; no natural-language mode in v1.
- `delete` must expose `--doc-only` and `--with-effects-if-available` modes.
- `render` is intentionally last because screenshot fidelity is the riskiest part.
- `apps/server/src/main.ts` currently only supports `serve` and `upgrade`, so canvas CLI work starts as a new command surface rather than extending an existing canvas command.
- The server/runtime can already open local state without the HTTP server: `apps/server/src/automerge-repo.ts` initializes a SQLite-backed Automerge repo in-process, and `packages/imperative-shell/src/database/db.ts` opens the same local SQLite database directly.
- That means doc/database inspection and mutation can plausibly run offline from CLI, without `serve`, as long as the operation only depends on persisted state.
- The current canvas runtime does **not** have a live document reconcile/subscription layer after startup. `SceneHydratorPlugin` loads once from `doc.groups` and `doc.elements`, and the spec explicitly calls out hydration as startup-only today.
- Consequence: external CLI edits to Automerge will not reliably appear in an already-open canvas UI until a reload or a future reconcile layer exists.
- Selection and camera state are runtime-local, not persisted. `CanvasService` stores `selection`, `focusedId`, `editingTextId`, and `editingShape1dId` in local Solid state, and the document model has no persisted camera/viewport fields.
- Consequence: a CLI command cannot truthfully answer “what the user is seeing right now” from persisted state alone. V1 likely needs explicit bounds/camera input or a documented “document-visible, not live-user-visible” contract.
- Groups are first-class in the document, but `TGroup` only stores identity, parent, order, lock, and creation metadata. Group bounds are derived at runtime from child nodes, not persisted.
- Consequence: CLI queries that mention group geometry or visible frames need a computed geometry layer, not just raw document reads.
- Groups also do not persist their own world position. In the current runtime, dragging a group patches descendant elements rather than a stored group `x` / `y` field.
- Consequence: a CLI `move group` operation is really a subtree-translation operation over descendant elements, not a single group-position patch.
- The current persisted element model is broader than the canvas spec summary alone: the codebase currently includes `rect`, `ellipse`, `diamond`, `arrow`, `line`, `pen`, `text`, `image`, `filetree`, `terminal`, `file`, and `iframe-browser` elements.
- Existing mutate paths are heavily UI/plugin-owned:
  - `GroupPlugin.group()` / `ungroup()` preserve absolute child positions and patch CRDT after Konva reparenting.
  - `RenderOrderPlugin` owns front/back/forward/backward ordering and persists `zIndex`.
  - `SelectPlugin` owns delete semantics, including collapsing nested selections to delete roots and recording undo snapshots.
  - shape/widget serialization helpers are registered via plugin capabilities such as `createShapeFromTElement`, `toElement`, and `toGroup`.
- Consequence: CLI work should avoid copying large chunks of Konva/plugin behavior ad hoc. The maintainable path is likely to extract pure document/geometry helpers into shared non-UI modules and let both CLI and runtime call them.
- `ContextMenuPlugin` already exposes `Bring to front`, `Move forward`, `Move backward`, `Send to back`, `Group`, and `Ungroup`, which is a good vocabulary match for future CLI verbs.
- Hosted widgets and browser widgets are visible through `worldWidgetsRoot` DOM overlays, not only through Konva layers. `stage.toDataURL()`-style rendering captures the stage but not the mounted DOM widget bodies.
- Consequence: a headless screenshot path based only on Konva will miss terminal/filetree/file/browser contents, or at least capture them only as invisible host rects unless a separate compositing strategy is added.
- The test harness in `packages/canvas/tests/test-setup.ts` proves the canvas runtime can be booted in a controlled test DOM and that stage snapshots can be exported, so there is already a useful experimental path for render/screenshot investigation.
- Delete side effects are inconsistent today and matter for CLI design:
  - filetree delete calls `api.filetree.remove`, which removes the DB row in addition to deleting the canvas element;
  - image delete uses the injected `deleteImage` capability to remove stored file blobs;
  - terminal delete calls `terminalLogic.removeTerminal()`, which ultimately uses `PtyService.remove(...)`.
- `PtyService` is in-memory within the running server process, not persisted in SQLite. A standalone CLI process cannot clean up another process’s live PTY sessions unless it talks to the running server or a new shared control path is introduced.
- This strongly suggests two layers:
  1. a document-first offline CLI core for inspect/query/patch/move/group/ungroup/delete-doc/reorder;
  2. optional live side-effect adapters when the server is running and the command explicitly asks for effect cleanup.
- Likely v1 constraint: hosted/interactive widgets should be treated as positioned canvas items with metadata and delete/move/reorder semantics, not as interactive app surfaces that the CLI can operate inside.
- Suggested command design direction:
  - read/query commands default to compact summaries;
  - mutation commands accept structured JSON patches/ops instead of natural-language freeform edits;
  - destructive commands expose explicit modes such as `--doc-only` vs `--with-effects-if-available`.
- Human intervention will be needed for:
  - deciding whether offline terminal deletion is allowed to silently skip PTY cleanup;
  - approving screenshot fidelity expectations for hosted widgets;
  - deciding whether future live-viewport awareness is worth frontend/session work;
  - manual acceptance testing against a real running canvas before implementation is considered done.

## Technical requirements for the CLI interface
- Command model:
  - The top-level shape should be `vibecanvas canvas <command>`.
  - Readonly commands and mutation commands must be strictly separated in command naming and help text.
  - Every command must support stable machine-readable output via `--json`.
  - Every command must support `--help` with explicit flag and output-contract documentation.
- Target resolution:
  - Commands must support both `--canvas <id>` and `--canvas-name <name>`.
  - If both selectors are present, the CLI must either reject the combination or document a single deterministic precedence rule; ambiguity must never be silent.
  - If neither selector resolves uniquely, the command must fail with a clear error instead of guessing.
  - `inspect <id>` must resolve the target by exact id within the selected canvas.
- Scope model:
  - Shared selectors must have explicit semantics for ids, types, group, subtree, bounds, and viewport.
  - Selector composition must be documented; default behavior should be intersection unless a command explicitly says otherwise.
  - Readonly commands may support broader selectors, but mutation commands should bias toward explicit, narrow targeting.
- Query interface:
  - `query` must remain strictly readonly.
  - The interface must not rely on natural-language parsing.
  - Query input should support layered forms: structured flags, `--where`, and optional structured JSON via `--query`.
  - Ambiguous combinations such as `--where` plus `--query` must be rejected.
  - The return contract must define stable shapes for zero matches, one match, and many matches.
- Mutation interface:
  - `patch`, `create`, `move`, `group`, `ungroup`, `delete`, and `reorder` must be deterministic.
  - Mutation payloads should use structured JSON, supplied inline only when small and otherwise from file or stdin.
  - In `--json`, mutation commands must report exactly what changed.
  - Destructive commands must expose explicit modes such as `--doc-only` versus `--with-effects-if-available`.
- Output contract:
  - `summary`, `focused`, and `full` must each have precise field-level definitions.
  - JSON output should include stable top-level metadata such as command, canvas id, match counts, changed counts, warnings, and errors.
  - Stdout versus stderr behavior must be predictable and test-friendly.
- Safety / testability:
  - `--db` and `--config` must be parsed before importing stateful modules.
  - Commands must be runnable against isolated test sandboxes.
  - No command may silently fall back to dev/prod storage during tests.
  - Dry-run support should be considered for mutation commands.
- Rendering / visibility contract:
  - `render` must explicitly state whether it captures only Konva output or also hosted DOM widgets.
  - Viewport-based commands must document that live user viewport/state is unavailable unless explicitly supplied.
- Versioning / evolution:
  - Keep v1 minimal and explicit.
  - Prefer additive evolution over changing the semantics of existing flags.
  - Help text and JSON output examples should become part of the test surface.
- Open design questions:
  - whether `--canvas-name` should be allowed if names are not guaranteed unique
  - whether `inspect <id>` should search only selected canvas or all canvases when no canvas selector is given
  - whether `query` should default to `summary` or `focused`
  - whether mutations should support `--dry-run`
  - whether `render` should be in v1 or postponed behind experimental status

## Concrete implementation strategy
- Phase 1: land A5 first so `vibecanvas canvas ... --db <path>` is parsed before any config/database/automerge modules with import-time side effects. No CLI test may fall back to developer-local storage.
- Phase 2: land A6 next by creating `packages/canvas/tests/cli/` with helpers that create a fresh temp directory, a fresh SQLite path, seeded canvas fixtures, and a real CLI process runner for every test case.
- Phase 3: ship readonly commands first: A7 `list`, A8 `inspect`, A9 `query`. `list` is the discovery command and may run without a canvas selector; all other scene commands require `--canvas` or a uniquely resolving `--canvas-name`.
- Phase 4: ship mutation commands one by one with tests-first discipline: A10 `patch`, A11 `move`, A12 `group`, A13 `ungroup`, A14 `delete`, A15 `reorder`.
- Phase 5: ship A16 `render` last and keep it explicit that v1 captures the headless stage path, not live user viewport state; DOM-hosted widgets must either be documented as partial/excluded or reported with warnings.
- Definition of done for every follow-up task: add end-to-end tests under `packages/canvas/tests/cli/`, pass `--db` explicitly, assert stdout/stderr/exit code, and verify the stable `--json` contract.

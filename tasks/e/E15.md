# E15 - canvas UI extensions: sideloadable community widgets and ArrowJS exploration

https://arrow-js.com/

## Plan
- Reframe the canvas extension discussion around true sideloaded code, not reuse of the internal `HostedSolidWidget` implementation.
- Evaluate ArrowJS as a possible runtime for community-provided UI extensions that mount into canvas-managed DOM overlays.

## NOTES

### Requirement update
- The current `HostedSolidWidgetPlugin` is **our internal implementation**, not the community extension API.
- Community items should still get something similar in capability: a custom UI surface, a toolbar affordance, and access to approved host APIs.
- "Sideloadable" should be taken literally: an extension can arrive as external code and be loaded by the host without rebuilding core canvas.

This changes the design target.
The earlier idea of simply opening the existing hosted widget system for third parties is too coupled to our internal Solid components and hardcoded widget types.
We need a separate extension runtime/API.

If we come to a general solution to run sideloaded frontend components fast and securly, we can remodel our solidjs components 
(file, terminal, filetree) to be sideloaded components too. (offical extension like) so user can replace them with their own versions if they want.

### Current canvas architecture constraints
Relevant files inspected:
- `packages/canvas/src/services/canvas/Canvas.service.ts`
- `packages/canvas/src/plugins/shared/interface.ts`
- `packages/canvas/src/plugins/Toolbar/Toolbar.plugin.ts`
- `packages/canvas/src/plugins/HostedSolidWidget/HostedSolidWidget.plugin.tsx`
- `packages/canvas/src/plugins/HostedSolidWidget/HostedSolidWidget.mount.tsx`
- `packages/canvas/src/components/FloatingCanvasToolbar/toolbar.types.ts`

What the current runtime already gives us:
- a plugin system with lifecycle hooks and a shared context
- a world DOM overlay root (`worldWidgetsRoot`) aligned to canvas coordinates
- CRDT-backed persisted elements
- injected host capabilities (`file`, `filetree`, `terminal`, `notification`, image APIs)
- toolbar control and tool selection

What is currently too internal/hardcoded for community use:
- tool ids are a static TypeScript union
- toolbar entries are hardcoded constants
- hosted widget type handling is hardcoded around built-ins (`filetree`, `file`, `terminal`)
- hosted widget mount/rendering is Solid-specific and coupled to internal components
- extensions would currently need raw access to internal plugin context to do anything useful

So the right move is not "expose HostedSolidWidget".
The right move is "build an extension host beside it".

### ArrowJS findings
Sources reviewed:
- `https://arrow-js.com/`
- `https://arrow-js.com/api`
- `https://github.com/standardagents/arrow-js`

What stands out:
- Arrow is very small and DOM-first.
- Core primitives are minimal: `reactive`, `html`, `component`, plus `watch`.
- It can run without a build step.
- It uses tagged template literals instead of JSX.
- It appears intentionally positioned as good for agent-authored UI code.
- The repo/docs also mention `@arrow-js/sandbox`: QuickJS/WASM-backed sandbox runtime for executing Arrow code off the host `window` realm while rendering into the real DOM.

That last point is the most interesting part for this problem.

### Why ArrowJS is attractive here
For sideloaded community extensions, the ideal UI runtime has these properties:
- very small API surface
- easy to learn from examples
- no required compile step
- can be shipped as plain JS modules
- easy to mount into an arbitrary DOM node
- works well for small self-contained widgets
- ideally supports some form of sandbox or isolation story

Arrow scores well on most of these.

Compared to exposing Solid to extension authors:
- Arrow has a much smaller surface area.
- Arrow can work with plain modules and template literals.
- Arrow feels more appropriate for externally supplied, self-contained widgets.
- Arrow reduces the need to share the app's exact build pipeline and JSX toolchain with extension authors.

Compared to framework-less custom elements only:
- Arrow gives state + declarative rendering without forcing us to invent our own tiny reactive layer.
- It is still much lighter than introducing a large framework runtime for extension code.

### But ArrowJS does not solve the whole problem by itself
Arrow helps with the widget frontend runtime.
It does **not** answer:
- how extensions are discovered and installed
- how permissions/capabilities are granted
- how toolbar items are contributed
- how CRDT element schemas are versioned and migrated
- how missing/crashed extensions render fallback UI
- how untrusted code is isolated

So Arrow should be thought of as a candidate **widget runtime**, not the whole extension architecture.

### Recommended architecture direction

#### 1. Separate host runtime from extension runtime
Host keeps ownership of:
- canvas placement and transforms
- selection/focus semantics
- document persistence
- z-order and deletion
- permissions/capabilities
- toolbar composition
- install/enable/disable state

Extension owns:
- widget UI DOM inside a mount root
- local ephemeral state
- explicit updates to its persisted element payload through host APIs
- optional use of approved host capabilities

#### 2. Extension contributions should be registry-driven
Need a host-side registry for:
- toolbar items
- extension element types
- widget mount factories
- optional commands/context menu items later

Likely shape:
- extension manifest
- JS module entrypoint
- host registration API

Example direction:
```ts
registerExtension({
  id: 'community.todo',
  version: '0.1.0',
  tools: [...],
  widgets: [...],
})
```

#### 3. Community widgets should mount into dedicated overlay roots
Keep the same broad rendering model as hosted widgets:
- canvas owns a positioned wrapper div
- extension renders inside that wrapper
- host controls mount/unmount/reposition
- extension does not manage canvas transforms itself

This keeps DOM overlays aligned with the existing canvas model.

#### 4. Keep the extension API narrow
Do **not** hand community code raw `IPluginContext`.
Instead expose a stable host bridge.

Rough direction:
```ts
type ExtensionHostApi = {
  element: {
    get(): TExtensionElement
    patch(next: Partial<TExtensionElementData>): void
    remove(): void
  }
  ui: {
    setTitle(title: string): void
    setSize(size: { width: number; height: number }): void
    requestFocus(): void
  }
  capabilities: {
    file?: ...
    filetree?: ...
    terminal?: ...
    notification?: ...
  }
}
```

This gives compatibility room later.

#### 5. Toolbar contribution should be metadata-first
Do not let extensions own arbitrary toolbar chrome in v1.
Let them contribute:
- id
- label
- icon
- shortcut maybe
- create behavior

Host still renders the toolbar.
That keeps UX coherent and prevents extensions from breaking the shell.

#### 6. Start with trusted sideloading, keep sandboxing on the design path
There are really two operating modes:

- **trusted/local sideloading**: load JS module into the main realm; easiest v1
- **sandboxed/community sideloading**: run in isolated realm/iframe/QuickJS-style sandbox with a message bridge

Arrow is interesting because its sandbox story may help the second mode.
But the first mode is still the shortest path to proving the API.

### Proposed v1 scope
v1 should support:
- install/load extension manifests + JS modules
- one toolbar item per extension widget type
- click-to-create extension widget element
- DOM overlay rendering into a host mount point
- persisted CRDT data owned by the extension element
- approved access to existing host APIs
- fallback shell when extension is missing or errors

v1 should avoid:
- arbitrary tapping into low-level canvas hooks
- direct access to raw Konva/stage/plugin internals
- custom extension-drawn Konva tools
- arbitrary replacement of host toolbar or shell UI

This keeps the surface understandable.

### ArrowJS fit assessment
My current assessment:

#### ArrowJS is a strong candidate when:
- widgets are small, self-contained, DOM-heavy apps
- extensions may be authored by humans or agents
- we want plain-module distribution
- we want to avoid forcing Solid as public extension API
- we may later need sandbox-friendly execution

#### ArrowJS is weaker if:
- we want deep integration with existing Solid component trees
- we want extension authors to reuse lots of app-internal UI primitives
- we want highly complex app-scale frontends with many existing ecosystem assumptions

For the stated problem, those weaknesses seem acceptable.
The extension runtime should be decoupled from our internal Solid host anyway.

### Recommendation
- Keep `HostedSolidWidget` internal.
- Design a new `ExtensionWidgetHost` for sideloaded community UI.
- Use a registry + manifest + host bridge architecture.
- Evaluate ArrowJS as the preferred default frontend runtime for extension widgets.
- Prototype v1 with trusted sideloaded modules first.
- Keep sandboxed execution as a first-class next step, especially if Arrow's sandbox package is viable in practice.

### Open questions
- Should extension UI run in the main realm at all, or must community code be sandboxed from day one?
- Is Arrow sandbox mature enough for production use, or do we need iframe isolation first?
- What is the extension packaging format: URL manifest, npm package, local folder, signed bundle?
- How should extension element data be namespaced and migrated?
- Can one extension contribute multiple widget types/tools?
- How much of the existing host capabilities should be exposed, and with what permission prompts?

### Follow-up implementation slices
1. Refactor toolbar/tool definitions to become registry-driven.
2. Introduce an extension registry and host bridge in `packages/canvas`.
3. Build a generic DOM overlay extension host separate from `HostedSolidWidget`.
4. Support local/trusted sideloaded module loading.
5. Add extension fallback/error shells.
6. Run a spike with one ArrowJS-powered widget.
7. Decide sandbox strategy and permissions model.

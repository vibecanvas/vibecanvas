## NOTES

### Current state
Raw canvas editing today is peer-style Automerge sync.

Path:
- frontend mutates local doc via `DocHandle.change(...)`
- frontend syncs raw Automerge messages to `/automerge`
- cli server websocket plugin forwards bytes into Automerge repo adapter
- Automerge repo merges and persists accepted CRDT changes

Files looked at:
- `packages/automerge-service/src/AutomergeServer.ts`
- `packages/automerge-service/src/adapters/websocket.adapter.ts`
- `apps/cli/src/plugins/automerge/AutomergePlugin.ts`
- `packages/canvas/src/services/canvas/crdt.ts`
- `packages/canvas-cmds/src/cmds/tx.cmd.patch.ts`

What is missing on raw `/automerge` path:
- runtime schema validation for `TCanvasDoc`
- size limits
- auth/authz
- app side effects hooks
- reject protocol back to clients

Important nuance:
- command APIs in `packages/canvas-cmds` do have some manual validation
- main canvas editor does not use that path for normal scene writes
- TypeScript types are compile-time only here, not runtime enforcement

### Target behavior
Need this:
- client writes locally first for speed
- UI updates immediately
- client syncs change to server
- server can inspect every incoming client sync message
- server may accept or reject based on:
  - invalid shape
  - too big
  - auth/authz
  - later: side-effect policies
- if accepted, normal broadcast continues
- if rejected, clients get explicit notification and refetch clean canvas state

This means server must be authority for admission.
But client still needs optimistic local state.

### Design constraint
Do not switch to server-first writes for every drag/key stroke.
Too slow.
Need optimistic client writes plus authoritative server admission.

### Main architecture options

#### Option A - server-first commands only
Pros:
- simplest authority model
- validation point obvious

Cons:
- breaks fast local editing requirement
- too much latency for drag/text editing

Not good for this problem.

#### Option B - keep raw Automerge peer sync, add interception/admission gate
Pros:
- keeps local optimistic writes
- preserves Automerge transport model

Cons:
- hard part is validation at change/message level
- reject path must unwind client local optimistic state
- side effects and auth now sit in low-level sync path

This is closest to requirement.

#### Option C - hybrid: optimistic client shadow doc + server canonical doc
Client edits local shadow doc immediately.
Server validates ops/intents and mutates canonical doc.
If accepted, canonical sync converges.
If rejected, client resets shadow from server.

Pros:
- clean authority model
- easier business validation than raw change inspection

Cons:
- more machinery
- likely bigger rewrite

Good long-term shape. Heavy for first move.

### Recommended direction
Take Option B first.
But structure it so it can evolve toward Option C later.

Concretely:
1. Intercept all inbound `/automerge` client messages before passing to repo.
2. Decode enough to identify:
   - sender peer
   - target document
   - proposed changes
3. Run server hook chain per incoming change batch:
   - auth hook
   - size hook
   - schema/business validation hook
   - side-effect hook
4. If accepted:
   - forward to Automerge repo normally
   - let repo persist and broadcast
5. If rejected:
   - do not forward message into canonical repo
   - send explicit reject event to sender
   - optionally notify other peers if needed
6. Client on reject:
   - freeze or mark canvas dirty/rejected
   - drop local handle state for that doc
   - refetch canonical doc from server storage
   - rebind canvas from clean state

### Server interception point
Best current hook point is near:
- `apps/cli/src/plugins/automerge/AutomergePlugin.ts`
- `packages/automerge-service/src/adapters/websocket.adapter.ts`

Today flow is basically:
`wsMessage -> adapter.message() -> receiveMessage() -> emit('message') -> repo`

Need a new boundary before repo consumes client changes.

Likely shapes:

#### Shape 1 - validating websocket adapter
Wrap existing websocket adapter with app hooks.

Pseudo shape:
```ts
wsMessage -> decode -> classify
         -> if sync/change message: validate(message, context)
         -> if ok: forward to repo
         -> if reject: send reject event
```

Pros:
- smallest surface change
- all raw `/automerge` traffic passes one place

Cons:
- validation logic works on Automerge protocol/change payloads
- business-level reasoning may be awkward

#### Shape 2 - validating repo/network bridge
Keep socket adapter thin.
Insert a bridge between network adapter events and repo.

Pros:
- cleaner separation of transport vs admission

Cons:
- depends on Automerge repo internals/events
- may be trickier with current library seams

For v1, Shape 1 looks more practical.

### Validation strategy
Need two layers.

#### Layer 1 - cheap admission checks on message/change envelope
Run before expensive materialization.
Examples:
- max websocket message bytes
- max changes per message
- max ops per change
- peer allowed for doc?
- sender authenticated?

This catches obvious abuse early.

#### Layer 2 - semantic document validation
Need to know whether applying the proposed change would create invalid canonical state.
Examples:
- bad canvas shape
- invalid element type fields
- too many elements
- giant image/base64 payloads
- broken parentGroupId
- group cycles
- future authz rules

Likely need a per-doc validation function like:
```ts
validateCanvasTransition({ beforeDoc, proposedDoc, peer, changeMeta })
```

That implies materializing candidate next state somewhere.

### Hard problem: how to validate raw Automerge changes safely
Server likely needs to apply incoming changes to a temporary candidate doc, not canonical doc first.
Then validate candidate.

Conceptually:
```ts
load canonical doc
clone/fork temp doc
apply incoming changes to temp doc
run validators on temp doc and transition
if ok: apply to canonical repo
if reject: discard temp doc
```

Need to confirm Automerge library seams support this efficiently.
This is the main technical unknown.

### Reject protocol
Need explicit app-level message back to client.
Raw silence is bad.
Client must know local optimistic state lost.

Need reject payload like:
```ts
{
  type: 'canvas-sync-rejected',
  docId,
  reason: {
    code: 'DOC_INVALID' | 'DOC_TOO_BIG' | 'FORBIDDEN' | 'AUTH_REQUIRED',
    message: string,
  },
  serverVersionHint?: string,
}
```

Client behavior:
- surface toast/banner
- stop trusting current local handle for that doc
- dispose and reload doc from server canonical storage
- rerender from clean state

### Why refetch is acceptable
Reject should be rare.
Normal path stays fast and local.
When reject happens, paying resync/refetch cost is fine.
This keeps normal UX snappy and failure path explicit.

### Side effects
User wants future side effects from admission path.
That fits if validation pipeline is a hook chain, not one giant validator.

Good hook groups:
- `beforeSyncMessage`
- `authorizeSyncMessage`
- `validateSyncMessage`
- `afterAcceptedSyncMessage`
- `afterRejectedSyncMessage`

Keep side effects only after accept unless explicitly needed.
Do not make validation phase mutate outside world.

### Risks
- Automerge protocol internals may make interception awkward.
- Temporary application of changes may be expensive on huge docs.
- Rejecting one peer after local optimistic apply needs careful client reset logic.
- If other peers already saw speculative local state through local-only UI assumptions, UX may flicker on reset.
- Need clear separation between protocol invalid, business invalid, and forbidden.

### Recommended v1 scope
- intercept inbound `/automerge` messages
- hard byte-size caps
- per-doc auth stub
- materialize candidate next canvas state for validation
- validate basic canvas invariants
- send explicit reject event to sender
- client full refetch on reject

### Later scope
- richer authz
- better partial rollback than full refetch
- server-side policy plugins
- metrics/audit logs for rejected sync
- move toward canonical-doc plus shadow-doc model if raw change validation gets too ugly

### Conclusion
Current system is fast but not authoritative.
Needed next step is not server-first writes.
Needed next step is a server admission layer in front of canonical Automerge apply.
That gives:
- optimistic local UX
- authoritative server reject/accept
- room for auth, size limits, and side effects
- clean fallback: reject -> notify -> refetch canonical state

## Executive summary
Big thing first.

There is not one bottleneck.
There are at least two heavy layers:
1. server does expensive synchronous filesystem scans
2. client renders too much tree UI at once and reacts globally

Most likely main cause for slow initial filetree load:
- `filesystem.files` recursively walks the directory tree up to `max_depth: 5` for every load/refetch
- it uses synchronous fs calls (`readdirSync`, `statSync`, `existsSync`)
- widget renders all returned rows recursively with no virtualization

Most likely main cause for slow path dialog open:
- dialog always does fresh `filesystem.list` call on open
- API lists and sorts all entries in target folder
- dialog renders every returned subfolder row with no virtualization or pagination
- dialog does not reuse filetree data or cache previous folder results

Watcher note:
- watcher is **not** part of first render path
- first render depends on `canvas.get` -> `filesystem.files` -> client render
- watcher matters after initial load, when folder contents change

Most likely main cause for "gets slow again" during active folder changes:
- file watch events refetch the whole tree every time
- refetch has no throttle/debounce/coalescing
- each refetch repeats recursive scan and rerender

## End-to-end flow

### Filetree widget load path
1. `FiletreeWidget` creates filetree logic.
2. `createFiletreeContextLogic` loads canvas data to find filetree row.
3. when `currentPath` is known, `treeData` resource calls `fetchTreeData()`.
4. `fetchTreeData()` calls `safeClient.api.filesystem.files({ path, glob_pattern, max_depth: 5 })`.
5. server `api.files-filesystem.ts` recursively walks directory tree.
6. client receives nested tree and recursively renders it in `renderTree()`.
7. file watch starts for current path.
8. every watch event triggers `refetchTreeData()` of whole tree.

### Path dialog load path
1. opening dialog triggers `createEffect` in `PathPickerDialog`.
2. it loads `initialPath` if present, else home dir then directory.
3. `loadDirectory()` calls `safeClient.api.filesystem.list({ path, omitFiles: true })`.
4. server reads all entries, filters, maps, sorts, returns all child folders.
5. dialog renders one button per child folder.
6. every open does fresh load. no cache.

## Code findings

### 1) Initial tree fetch is recursive and wide
Evidence:
- `createFiletreeContextLogic.ts:98-105`
- always calls `filesystem.files` with `max_depth: 5`

```ts
const [listError, listResult] = await args.safeClient.api.filesystem.files({
  query: {
    path,
    glob_pattern: globPattern ?? undefined,
    max_depth: 5,
  },
});
```

Why this matters:
- for large roots, this is not "list current folder"
- this is "walk subtree up to 5 levels deep"
- cost grows with total descendants, not just visible top-level rows

Extra detail:
- server logic still includes entries when `depthRemaining === 0`
- so practical breadth at deepest included level is still large
- this makes `max_depth: 5` more expensive than a shallow single-level listing

### 2) Server walk is synchronous and blocking
Evidence:
- `api.files-filesystem.ts:21` calls recursive `walkDirectory`
- `FilesystemServiceNode.ts:29-37` uses `readdirSync`
- `FilesystemServiceNode.ts:41-49` uses `statSync`
- `FilesystemServiceNode.ts:25-27` uses `existsSync`

Why this matters:
- large scan blocks server event loop
- while scan runs, this process cannot cheaply serve other work
- slow scan becomes UI latency and maybe general app stall

### 3) Server recursively sorts every directory
Evidence:
- `api.files-filesystem.ts:38`
- `api.list-filesystem.ts:15-22`

Why this matters:
- each directory read allocates arrays
- each array is sorted
- on many directories this becomes repeated `O(n log n)` work
- on giant flat folders, single sort alone can cost enough to feel slow

### 4) Glob filtering recompiles regex for every entry
Evidence:
- `api.files-filesystem.ts:45`, `52`, `59-62`
- `globMatch()` builds a new `RegExp` every call

Why this matters:
- when glob is active, every entry pays regex compilation cost
- this is avoidable CPU churn
- likely not the main cause with empty glob
- likely meaningful when glob is used on large trees

### 5) Widget renders full returned tree, no virtualization
Evidence:
- `FiletreeWidget.tsx:47-105` recursive `renderTree`
- `FiletreeWidget.tsx:177-180` renders all root children
- child folders render all open descendants

Why this matters:
- large payload means many buttons, icons, nested wrappers, reactive reads
- every row binds click and drag handlers
- there is no viewport windowing
- DOM size and JS setup both scale with visible row count

### 6) Row state is driven by global signals read by many rows
Evidence:
- `FiletreeWidget.tsx:48-50`
- every row reads:
  - `openFolders()`
  - `selectedRowPath()`
  - `dragOverTargetPath()`

Why this matters:
- toggling one folder clones the whole `Set`
- every rendered row checks membership again
- selecting one row can re-run row-level reactive work across many rows
- drag-over state can churn while pointer moves

This may not dominate initial load, but it can dominate interaction cost on large visible trees.

### 7) Lazy folder expansion is not truly shallow
Evidence:
- `createFiletreeContextLogic.ts:290-304`
- `loadSubdirectoryChildren(folderPath)` calls same `fetchTreeData()`
- `fetchTreeData()` still requests `max_depth: 5`

Why this matters:
- opening one folder can scan five levels under that folder
- user expects "expand one level"
- current behavior can fetch much more than needed

### 8) Replacing lazy-loaded children traverses existing tree
Evidence:
- `createFiletreeContextLogic.ts:266-303`
- `replaceNodeChildren()` recursively searches tree to replace one folder

Why this matters:
- every expand on a large already-loaded tree walks existing tree structure
- then Solid gets a new top object and may update broad subtree
- secondary cost, but real

### 9) Watch events refetch whole tree with no coalescing
Evidence:
- `createFiletreeContextLogic.ts:165-170`

```ts
for await (const _event of iterator) {
  if (abort.signal.aborted || activeWatchId !== watchId) break;
  lazyLoadedFolderPaths.clear();
  void refetchTreeData();
}
```

Why this matters:
- one save can emit multiple fs events
- build tools can emit many events fast
- every event triggers full tree fetch again
- on large roots this can create event storm -> fetch storm -> rerender storm

### 10) Path dialog always cold-loads current folder contents
Evidence:
- `PathPickerDialog.tsx:70-82`
- every open calls `loadDirectory(initialPath)` or `loadHomeDirectory()`
- there is no memoized cache by path

Why this matters:
- even if widget already loaded same path tree, dialog does not reuse it
- first open after startup is guaranteed remote call + full list + full render
- repeated opens still do same work

### 11) Path dialog renders all child folders with no limit
Evidence:
- `PathPickerDialog.tsx:157-176`
- `api.list-filesystem.ts:15-29` returns all children

Why this matters:
- folder with thousands of subfolders means thousands of buttons
- even with `omitFiles: true`, large dev roots can still have many dirs
- dialog does not page, search, or virtualize

### 12) Path dialog load does duplicate filtering
Evidence:
- server already filters when `omitFiles: true` in `api.list-filesystem.ts:15-16`
- client filters again at `PathPickerDialog.tsx:51`

Why this matters:
- small waste only
- not root cause
- mention for completeness

## Synthetic measurements
These are not production traces.
These are small local measurements to test algorithm shape.
Still useful.

### Experiment A - flat directory listing vs recursive tree walk
Setup:
- temp dir with 2,000 subdirs and 8,000 files
- benchmark code mirrors current API behavior

Results:
- `list(..., omitFiles: true)` over 2,000 returned dirs: **11.69 ms**
- `list(..., omitFiles: false)` over 10,000 entries: **14.51 ms**
- recursive `walkDirectory(..., maxDepth: 5)` over same flat root: **77.46 ms**
- recursive `walkDirectory(..., maxDepth: 5, '*.txt')`: **67.83 ms**

Interpretation:
- recursive tree fetch is much more expensive than one-level list
- flat path dialog cost can already be noticeable with very large folders
- tree widget initial load cost is materially worse than dialog cost

### Experiment B - nested tree recursive walk
Setup:
- synthetic tree depth 5
- each dir has 5 subdirs and 10 files
- total returned nodes: 11,715

Results:
- recursive walk without glob: **168.20 ms**
- recursive walk with `*.txt`: **195.67 ms**

Interpretation:
- nested tree makes current recursive scan clearly expensive
- this is before network transport and before DOM render
- UI-visible latency can easily be much higher end to end

### Experiment C - glob regex compilation overhead
Setup:
- 200,000 filename checks
- compare current `globMatch()` behavior vs precompiled regex

Results:
- current behavior: **86.97 ms**
- precompiled regex: **6.98 ms**
- speedup: **12.46x**

Interpretation:
- glob regex recompilation is a real CPU tax
- likely secondary to recursive walking, but worth validating in real traces
- especially relevant when glob filtering is active on large roots

## Bottleneck ranking

### Rank 1 - recursive subtree fetch on widget load
Confidence: high
Impact: high

Reason:
- directly in hot path
- always happens for widget load
- cost scales with descendants
- measured expensive in synthetic benchmark

### Rank 2 - full tree render with no virtualization
Confidence: high
Impact: high

Reason:
- all returned nodes become DOM/reactive work
- big subtree payload turns into many rows and handlers
- likely visible jank after API already paid the scan cost

### Rank 3 - watch-triggered whole-tree refetch storm
Confidence: high
Impact: high in active repos

Reason:
- direct code evidence
- file watchers often emit bursts
- cost repeats rank 1 and rank 2 many times

### Rank 4 - path dialog lists and renders all child dirs on every open
Confidence: high
Impact: medium to high depending on directory width

Reason:
- fresh load each open
- all subfolders returned and rendered
- no cache, no reuse, no virtualization

### Rank 5 - synchronous filesystem API
Confidence: high
Impact: medium to high

Reason:
- blocking cost likely amplifies perceived latency
- hurts server responsiveness under concurrent work
- may matter more in desktop app with one shared process

### Rank 6 - global signal fan-out across many rows
Confidence: medium
Impact: medium

Reason:
- very plausible interaction cost source
- needs browser profiling to size precisely

### Rank 7 - glob regex compile per entry
Confidence: high
Impact: low to medium overall, high when glob active

Reason:
- measured in microbench
- only active when glob is used
- probably not main default complaint unless users use glob often

### Rank 8 - replaceNodeChildren full-tree traversal on expansion
Confidence: medium
Impact: low to medium

Reason:
- real cost on expand
- but likely smaller than scan + render

## Hypotheses
Need more explicit hypothesis section.
Each hypothesis should be tested in isolation before fix work.

### H1 - first-render slowness is mostly in `api-filesystem/files`
Claim:
- first render is slow because `filesystem.files` recursively walks too much data
- watcher is not involved in first render

Why plausible:
- widget calls `filesystem.files({ max_depth: 5 })` on first load
- API recursively walks tree before UI can render
- synthetic bench shows recursive walk cost grows fast with tree size

What would falsify it:
- API time is small, but browser render time is huge
- or RPC payload/deserialize time dominates

### H2 - client render cost is large after API returns
Claim:
- even if server returns in acceptable time, filetree still feels slow because it renders too many rows and handlers

Why plausible:
- recursive DOM render
- no virtualization
- many row-level reactive reads

What would falsify it:
- profiler shows render cost is tiny relative to API wait

### H3 - path dialog slowness is independent of watcher and mostly caused by `filesystem.list` + full render
Claim:
- path dialog is slow because it cold-loads and renders all child directories on every open

Why plausible:
- dialog does fresh `list` request every open
- no cache
- no row limit or virtualization

What would falsify it:
- API and render both small on large folders
- or slowness only appears when widget watcher is already active

### H4 - watch storms are a secondary problem, not first-render problem
Claim:
- watcher does not explain initial open, but does explain repeated slowness after files change

Why plausible:
- watch starts after `currentPath` is set
- each event triggers full `refetchTreeData()`

What would falsify it:
- no event bursts in real repro
- or refetches are negligible compared to idle state

### H5 - sync filesystem service amplifies API cost but is not sole root cause
Claim:
- `packages/service-filesystem` contributes because it executes reads synchronously
- but API shape still matters more than sync vs async alone

Why plausible:
- recursive walk + sync `readdirSync` blocks server thread
- but asking for too much work would still be expensive even async

What would falsify it:
- isolated package benchmark shows sync layer cost is tiny relative to higher-level overhead

### H6 - glob cost is real but not dominant for default no-glob case
Claim:
- glob regex compilation wastes CPU when glob is active
- but removing glob alone will not solve big-folder slowness

Why plausible:
- microbench showed large regex compile overhead
- default complaint exists even without glob usage

What would falsify it:
- real traces show glob path dominates in most repros

Most likely stack:

- So for that trace, the original startup-hydration hypothesis was **not** the dominant cause.
- The second trace strengthened that further:
  - boot still stayed fast
  - but the runtime trace window reported multiple long tasks and frame gaps
  - including a ~`3608ms` long task and a matching ~`3600ms` frame gap shortly after boot
- The third trace kept the same pattern:
  - boot was still fast (`find-document-ready` ~`92.8ms`, total runtime boot ~`23ms`, init hydrate ~`9.4ms`)
  - but post-boot long tasks still dominated (`109ms`, `105ms`, `344ms`, `448ms`, `3543ms`, `929ms`)
  - there were still matching frame gaps (`333ms`, `441.7ms`, `3533.4ms`, `920.8ms`)
  - and there were **no `plugin:image` logs** in the pasted output, which weakens the image-decode theory for this run
- A follow-up manual experiment also ruled out the strongest devtools theory:
  - disabling `solid-devtools` import/plugin did **not** remove the freeze
  - the camera still could not move immediately
  - the user does **not** have the Solid Devtools browser extension installed
- That means the issue is now much more likely one of:
  - a later long task / frame stall after boot
  - a visually blocked repaint even though camera state updates
  - another frontend-side mounted workload outside the measured canvas boot/hydrate path
  - especially either:
    - a hosted widget path such as terminal, filetree, or file/PDF viewer startup
    - or the frontend shell/sidebar store path if many canvases are being rendered in the app chrome
- Latest field data weakened both of those narrower suspects:
  - the user has only `8` canvases in the sidebar
  - `canvas.list()` completed in ~`186.3ms`
  - the affected canvas has **no hosted widgets**
  - the freeze affects **both** wheel pan and hand drag
- So the remaining leading suspects are now:
  - canvas render/draw cost after boot, especially Konva layer draws triggered once the scene is mounted
  - canvas chrome overlays/plugins mounted around the runtime, such as toolbar, selection-style-menu, context-menu, recorder, or visual-debug
  - dev-mode / HMR / Solid-refresh overhead outside the app-specific startup timings we already measured

## Investigation snapshot
Current startup sequence of interest:
1. `Canvas.tsx` resolves a doc handle through `findDocument(url)`.
2. `buildRuntime(...)` is created only after that handle exists.
3. Runtime boot calls `hooks.init.call()` and then awaits `hooks.initAsync.promise()`.
4. The only current `initAsync` canvas plugin is `scene-hydrator`.
5. `scene-hydrator` synchronously reads the full doc, sorts all groups/elements, mounts nodes, sorts scene order, and batch-draws the stage.
6. On non-local CRDT changes it can still destroy foreground children and fully rebuild again.

So the leading architectural suspicion is:
- **input exists conceptually, but the browser main thread is too busy during startup to process camera interaction promptly**.

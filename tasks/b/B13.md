## NOTES
- Category: bug.
- Reproduced from the real dev environment with `bun run apps/cli/src/main.ts ...`, not the isolated test harness.
- `canvas list --json` returned 5 canvases from `local-volume/data/vibecanvas.sqlite`.
- `canvas query --canvas-name hello --json` works.
- `canvas query --canvas-name ok --json` fails with `CANVAS_QUERY_FAILED` / `Document automerge:... is unavailable`.
- The same unreadable-doc failure also reproduces for `power`, `test ground`, and `recorder`.
- Direct SQLite inspection shows the root of that split clearly:
  - the `canvas` table contains 5 rows (`hello`, `power`, `test ground`, `recorder`, `ok`);
  - `automerge_repo_data` contains storage entries for exactly one Automerge document id, the one behind `hello` (`2mRqoMhFziiahBTTUDYDNdY7w9oU`);
  - there are no persisted Automerge chunks for the Automerge ids behind `power`, `test ground`, `recorder`, or `ok` in `local-volume/data/vibecanvas.sqlite`.
- So `list` works because it only reads SQL rows, but `query`/`move`/`patch` fail for those canvases because the current `apps/cli/src/canvas-cli/local-state.ts` implementation is offline-only: it opens a fresh SQLite-backed Automerge repo, calls `repo.find(row.automerge_url)`, and throws if `handle.doc()` is missing. There is no live websocket fallback in this codepath.
- That means the real bug is a regression / divergence between the new shared CLI implementation in `apps/cli/src/canvas-cli/local-state.ts` and the older server-side scene CLI path documented in A11. The server-side path (`apps/server/src/canvas-cli/offline-state.ts` + `scene-shared.ts`) already falls back from offline SQLite Automerge access to a live `/automerge` websocket repo when the SQL row exists but the local SQLite Automerge chunks do not.
- `hello` is queryable only because it is the one canvas whose Automerge document is actually present in the local SQLite Automerge storage. The other listed canvases are unreadable because their rows survived in SQL but their Automerge docs are only available from the running live repo, not from the offline SQLite file that the new CLI code reads.
- `canvas patch` works on `hello`, but at least twice returned `CANVAS_PATCH_FAILED` with `Timed out waiting for persisted canvas doc ...` even though an immediate follow-up `query` showed the mutation had actually landed.
- One real repro of the false-negative patch path was adding a new style key such as `strokeColor` to an existing rect.
- That second issue is not the same root cause as the unreadable-doc failure. The patch timeout happens on `hello`, which *is* readable offline, so the Automerge-doc-missing problem is not involved there.
- The patch false-negative root cause is the equality check in `packages/canvas-cmds/src/cmd.patch.ts`: `areValuesEqual()` uses `JSON.stringify(left) === JSON.stringify(right)`. Patch verification waits until a reread doc is byte-for-byte string-equal to the expected object graph. When a patch adds nested keys (for example a new `style.strokeColor` key), Automerge can persist an equivalent object with different property insertion order, so the semantic mutation has landed but the stringified object still differs and `waitForMutation()` times out.
- So B13 currently contains two bugs:
  1. unreadable real canvases: missing live fallback in `apps/cli/src/canvas-cli/local-state.ts`, caused by the new CLI path not carrying over the A11 server-side fallback logic;
  2. false patch timeout on readable canvases: key-order-sensitive verification in `packages/canvas-cmds/src/cmd.patch.ts` via `JSON.stringify` equality.

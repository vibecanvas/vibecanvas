## NOTES
- Category: bug.
- User symptom: Ctrl+C in the canvas terminal shows `^C` repeatedly but does not terminate the running process; the same command in a normal OS terminal interrupts correctly.
- Likely not caused by canvas event hooks stealing input, because keyboard events inside hosted widgets are ignored by `packages/canvas/src/plugins/EventListener/EventListener.plugin.ts`.
- Frontend / transport path appears mostly pass-through:
  - `packages/canvas/src/components/terminal/createTerminalContextLogic.ts` sends terminal input over websocket.
  - `packages/canvas/src/services/terminal/pty.ts` forwards websocket payload.
  - `apps/server/src/server.ts` forwards PTY websocket messages into the PTY attachment.
  - `packages/imperative-shell/src/pty/srv.pty.ts` creates `new Bun.Terminal(...)`, passes it into `Bun.spawn(..., { terminal })`, and injects input via `terminal.write(...)`.
- Ghostty OSC 12 / 112 warnings look unrelated and are more likely output parsing / cursor-color warnings than input or signal handling bugs.
- Strongest current root-cause hypothesis: Bun PTY / `Bun.Terminal.write("\x03")` is bypassing proper PTY line-discipline / controlling-terminal signal behavior.
- External upstream evidence:
  - Bun issue `oven-sh/bun#25779` — `Bun.Terminal.write() bypasses PTY line discipline - Ctrl+C doesn't generate SIGINT`.
  - Duplicate Bun issue `oven-sh/bun#25832` — `Bun.Terminal: Ctrl+C (\x03) does not send SIGINT to foreground process`.
  - Bun PR `oven-sh/bun#25834` — `fix(terminal): enable SIGINT/SIGTSTP signals for existing Terminal objects`.
  - Bun PR `oven-sh/bun#26008` — `fix(Terminal): generate signals for Ctrl+C/Ctrl+\\/Ctrl+Z`.
- Key observations from the Bun reports to preserve here:
  - `stty` can still report `isig`.
  - `^C` is echoed.
  - The foreground process is not interrupted.
  - The bug is reported on `darwin arm64` too.
- Local environment relevance: this repo currently uses Bun `1.3.10`, while the Bun PTY fix PRs above are still open.
- Manual confirmation gathered during this task:
  - hosted terminal `sleep 100` echoed `^C` while the process kept running.
  - hosted terminal `trap 'echo SIGINT_RECEIVED; exit 0' INT; echo READY; sleep 30` never printed the trap output and stayed blocked.
  - hosted terminal `cat | od -c` showed `^C` as literal input data.
  - hosted terminal `stty -a | grep -o '\-*isig'` still reported `isig`.
- Fix implemented:
  - `packages/imperative-shell/src/pty/srv.pty.ts` no longer uses `new Bun.Terminal(...)` plus `Bun.spawn(..., { terminal })`.
  - the PTY backend now uses `bun-pty` via `spawn(...)`, `onData(...)`, `onExit(...)`, `write(...)`, `resize(...)`, and `kill(...)`.
  - websocket attach, replay buffer, and PTY session metadata remained in `PtyService`; only the PTY backend implementation changed.
  - incoming websocket bytes are still passed through directly, now via the `bun-pty` process instead of Bun's broken terminal path.
  - environment variables are now normalized into `Record<string, string>` before spawning so the PTY backend does not receive `undefined` env values under strict TypeScript checking.
- Regression coverage added:
  - `packages/imperative-shell/src/pty/srv.pty.test.ts` now includes a Ctrl+C interrupt test that starts `sleep 30` behind a shell SIGINT trap, writes `\x03`, and asserts `SIGINT_RECEIVED` is emitted.
- Validation:
  - `bun test src/pty/srv.pty.test.ts` passes with 4 tests.

### Root-cause confirmation strategy
- If the frontend emits char code 3 and websocket/server preserve it unchanged, frontend transport is cleared.
- If `cat | od -c` inside the affected PTY shows Ctrl+C as literal input data, line-discipline signal generation is not happening.
- If `stty` still reports `isig`, terminal settings alone are not enough to explain the failure.
- If the minimal Bun repro fails outside the app while the same shell commands work in a normal OS terminal, the app is not the root cause.
- Together these observations confirm Bun PTY behavior as the likely root cause before we choose a workaround or replacement.

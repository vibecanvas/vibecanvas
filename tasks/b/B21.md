# B21 - filesystem: recursive filetree scan dies on EPERM/EACCES folders like `~/.Trash`
[Overview](../BASED.md)

Mac privacy and filesystem permissions can deny some subfolders. Filetree must skip denied paths and still show rest.

## Context
Relevant files:
- `packages/api-filesystem/src/api.files-filesystem.ts`
- `packages/api-filesystem/src/api.list-filesystem.ts`
- `packages/service-filesystem/src/FilesystemServiceNode.ts`

Current behavior:
- `FilesystemServiceNode.readdir()` wraps `readdirSync()` and returns one generic failure for any denied path.
- `api.files-filesystem.ts` recursively walks children.
- if one child dir fails, whole walk aborts.
- example on macOS: `EPERM: operation not permitted, scandir '/Users/omarezzat/.Trash'`

Likely cause:
- macOS TCC / privacy controls deny some folders even inside home dir.
- hidden/system folders like `~/.Trash` can be unreadable without Full Disk Access or explicit user grant.
- same class of failure can also show as `EACCES`.

Wanted behavior:
- inaccessible child folder does not kill whole tree.
- accessible siblings still load.
- UI still lists denied folders like `.Trash`.
- UI marks denied folder clearly, e.g. red row + hover text `Permission denied`.

## Plan
### 1. Classify permission failures
- map `EPERM` / `EACCES` from filesystem service into a distinct permission-denied error shape
- keep not-found and other failures separate

### 2. Make recursive walk tolerant
- in `walkDirectory()`, skip denied child directories instead of returning fatal error
- keep fatal behavior for root path failure
- decide whether denied nodes appear with empty children + metadata, or are omitted entirely

### 3. Add safe fallback surface
- return partial results cleanly
- expose enough metadata for UI/logs: skipped count, denied paths, or per-node `unreadable` / `errorCode` flag
- keep denied node in tree result instead of dropping it when parent listing succeeded
- avoid noisy hard failures when only one subtree is blocked

### 4. Test real cases
- add tests for denied nested dir, denied root dir, and mixed accessible/inaccessible siblings
- verify `list` and `files` APIs match chosen behavior

## NOTES
Good fallback rules:
- root requested path denied -> return explicit permission error
- child path denied during recursion -> skip child and continue
- file `stat` denied -> treat same as unreadable child, not full-tree failure
- watcher setup on denied path should fail only for that path

Simple first fix:
- only make recursive walk tolerant
- leave service sync for now
- no permission prompt system needed

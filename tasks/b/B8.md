# B8 - hosted widgets: transformer resize loses control when pointer crosses DOM

## Plan
- Confirm the intended interaction contract for hosted DOM widgets on the canvas.
- Compare iframe-browser and shared hosted-widget pointer-event gating during focus vs resize/transform states.
- Document the likely bug surface, reproduction path, and candidate fix directions before implementation.

## NOTES
- Category: bug.
- User intent: hosted DOM widgets should only receive pointer events while explicitly focused for interaction. When not focused, the canvas should keep panning/selection control.
- Reported regression: after focusing a hosted/iframe widget so its DOM becomes interactive, clicking the resize button should still allow dragging Konva transformer handles across the widget surface without the DOM stealing pointer events mid-gesture.
- Expected contract:
  - unfocused widget DOM: pointer events blocked so canvas pan/select works
  - focused widget DOM: pointer events enabled so widget internals work
  - transformer-visible / active resize state: widget DOM blocked again so resize drag can pass over the widget body without interruption
- Relevant runtime architecture from `docs/specs/spec.canvas.md`:
  - hosted widgets are invisible `Konva.Rect` hosts in `staticForegroundLayer`
  - visible DOM is projected through `worldWidgetsRoot`
  - `TransformPlugin` shows the shared `Konva.Transformer` only when hosted widgets set `vcHostedTransformerVisible=true`
- Code research:
  - `packages/canvas/src/plugins/Transform.plugin.ts:166-177` explicitly hides the transformer for single hosted selections unless `vcHostedTransformerVisible === true`.
  - `packages/canvas/src/plugins/IframeBrowserWidget.plugin.tsx:817-846` already computes `selectable`/`interactive` with `transformerVisible` awareness and disables mount + iframe pointer events while resize handles are shown.
  - `packages/canvas/src/plugins/HostedSolidWidget.plugin.tsx:756-772` currently gates DOM interactivity only on `focusedId === node.id()` and `mode === SELECT`, without checking `vcHostedTransformerVisible`.
  - This means shared hosted widgets (`terminal`, `filetree`, `file`) can stay DOM-interactive even after the resize button enables the transformer, which matches the reported regression path.
  - The iframe plugin appears to have the newer protection, while the shared hosted widget plugin still uses the older focus-only rule. That mismatch is likely why the bug resurfaced in one path while fixing the other.
- Existing test coverage:
  - `packages/canvas/tests/plugins/iframe-browser/IframeBrowserWidgetPlugin.test.ts:166-233` already asserts that iframe DOM pointer events turn off when the resize button shows transformer handles.
  - `packages/canvas/tests/plugins/hosted-widget/HostedSolidWidgetPlugin.test.ts:581-613` only asserts focus vs hand-mode interactivity and does not cover transformer-visible gating.
- Likely fix direction:
  - extract or replicate the iframe plugin's transformer-aware gating into `HostedSolidWidgetPlugin`
  - ensure root mount, header buttons, and hosted body all switch to `pointer-events: none` / `inert` while transformer handles are active
  - add a regression test mirroring the iframe resize gating test for terminal/filetree/file widgets
- Implemented fix:
  - added a regression test in `packages/canvas/tests/plugins/hosted-widget/HostedSolidWidgetPlugin.test.ts` that focuses a terminal widget, clicks the resize button, and asserts the hosted DOM becomes non-interactive while transformer handles are shown
  - updated `packages/canvas/src/plugins/HostedSolidWidget.plugin.tsx` so mount interactivity and `HostedWidgetShell.isInteractive()` both disable DOM pointer events when the selected node has `vcHostedTransformerVisible === true`
  - updated `packages/canvas/src/plugins/IframeBrowserWidget.plugin.tsx` so the browser DOM mount is only interactive when the widget is actually focused, instead of being selectable while unfocused
  - updated `packages/canvas/src/plugins/IframeBrowserWidget.plugin.tsx` again so a focus-click does not re-enable iframe DOM mid-gesture; the browser DOM only arms after the pending pointer is released
- Validation:
  - `bun run test -- tests/plugins/hosted-widget/HostedSolidWidgetPlugin.test.ts` passes
  - `bun run test -- tests/plugins/iframe-browser/IframeBrowserWidgetPlugin.test.ts` passes
  - test run still emits pre-existing CodeMirror/jsdom `getClientRects` stderr noise in unrelated hosted file tests, but the suite passes

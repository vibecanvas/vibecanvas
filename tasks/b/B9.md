# B9 - hosted widgets + iframe browser: canvas drag moves visually but does not persist on reload

## Plan
- Confirm the normal canvas drag flow for hosted and iframe widgets from the canvas spec and plugin code.
- Document the persistence gap between the native Konva drag path and the DOM header drag path.
- Capture test gaps and the likely fix direction in a dedicated bug task before implementation.

## NOTES
- Category: bug.
- User report: dragging a hosted element or iframe in the canvas appears to work, but after reload the element returns to the old position.
- Likely root cause: both plugins persist the custom DOM-header drag path via `beginDomDrag()`, but their normal Konva host rect path only syncs DOM on `dragmove transform` and does not patch CRDT on `dragend`.
- Relevant files:
  - `packages/canvas/src/plugins/HostedSolidWidget.plugin.tsx`
  - `packages/canvas/src/plugins/IframeBrowserWidget.plugin.tsx`
- Relevant code observations:
  - hosted plugin creates draggable `Konva.Rect` hosts and listens to `dragmove transform` only for projection sync
  - iframe plugin does the same
  - neither plugin registers a standard Konva `dragstart`/`dragend` persistence/history flow like `Shape2dPlugin`/`TextPlugin` do
  - this explains why the widget visibly moves in-session but reload rehydrates from stale document coordinates
- Test coverage gap:
  - existing tests cover DOM header drag persistence (`bridges header drag from DOM back into ...`) but not native Konva rect drag persistence
- Reproduction result:
  - added failing regression tests that move the native hosted/iframe `Konva.Rect` hosts through the canvas drag path, then recreate the canvas and assert the new position survives reload
  - both tests currently fail because the in-memory node position updates to the dragged coordinates, but the Automerge document stays at the original coordinates
- Likely fix direction:
  - add native hosted/iframe drag persistence on Konva `dragstart`/`dragend` (and multi-select/history if needed), or share the same persistence logic between DOM-drag and native-drag paths
- Implemented fix:
  - `HostedSolidWidget.plugin.tsx` now snapshots the widget element on native `dragstart`, writes CRDT on native `dragend`, and records undo/redo history for the native Konva drag path
  - `IframeBrowserWidget.plugin.tsx` now does the same for iframe browser widgets
- Validation:
  - `bun run test -- tests/plugins/hosted-widget/HostedSolidWidgetPlugin.test.ts tests/plugins/iframe-browser/IframeBrowserWidgetPlugin.test.ts` passes
  - the run still emits the pre-existing CodeMirror/jsdom `getClientRects` stderr noise from unrelated hosted file tests, but the suite passes

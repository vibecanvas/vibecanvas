## NOTES
- Category: bug.
- User report: select an element or group so the style menu appears, drag it to a new position, then change a style such as color, width, or opacity. The style change applies, but the node jumps back to the original position from when the style menu / transformer first appeared.
- Reproduction scope: reproduced across multiple stylable canvas element types, which suggests the bug is systemic within the selection style menu flow rather than isolated to one plugin.
- Selection style menu flow:
  - `packages/canvas/src/plugins/SelectionStyleMenu/SelectionStyleMenu.plugin.ts` computes `resolved = createMemo(() => this.getResolvedSelection(context))`.
  - style action handlers then call `this.applyStyleChange(context, resolved(), ...)`, so they can operate on a memoized resolved selection instead of re-reading live nodes at click time.
  - relevant action flow is around `packages/canvas/src/plugins/SelectionStyleMenu/SelectionStyleMenu.plugin.ts:71-95` and `:198-317`.
- Root cause hypothesis from code research:
  - `getResolvedSelection()` serializes the current selection into full `TElement` snapshots via `context.capabilities.toElement?.(node)`.
  - dragging changes live node position, but those `x/y/rotation/...` changes are not Solid dependencies of the selection memo as long as the selected node set stays the same.
  - that means `resolved()` can keep an older pre-drag full-element snapshot even after the user moves the selection.
  - `applyStyleChange()` then clones those stale full elements, mutates only style/data fields, and writes them back through `context.capabilities.updateShapeFromTElement?.(element)` plus `context.crdt.patch({ elements: dedupedAfterElements, groups: [] })`.
  - because the downstream update paths consume full element geometry, stale `x/y/rotation/...` from the memoized snapshot overwrite the live dragged position, causing the visible jump.
- Supporting code references:
  - `packages/canvas/src/plugins/Shape2d/Shape2d.plugin.ts:321-345` resets world position from `element.x/y`.
  - `packages/canvas/src/plugins/Shape1d/Shape1d.plugin.ts:910-923` resets world position from `element.x/y`.
  - `packages/canvas/src/plugins/Text/Text.plugin.ts:263-285` resets world position from `element.x/y`.
- Why it appears selective:
  - this is most visible on content types that expose the selection style menu.
  - hosted widgets / terminal likely do not reproduce through the same user flow only because they do not surface this style menu, not because their geometry update paths are inherently safe.
- Fix direction to preserve in implementation:
  - add regression coverage for drag-then-style on at least a rect and a selected group, asserting that style updates do not alter current position.
  - re-resolve the current selection from live nodes right before style application, or otherwise derive fresh geometry at patch-build time instead of relying on the cached memoized full-element snapshot.
  - keep style-menu writes limited to the attributes actually being changed so stale geometry is not written back through update/CRDT.
  - validate across shape2d, shape1d, text, and group selection flows.
- Current status: research only; no implementation has been done yet.

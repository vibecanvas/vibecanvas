## NOTES
- User report: with dev server on port 3000, running `vibecanvas canvas add ...` succeeds but frontend does not show the new shape until page refresh.
- `apps/cli/src/plugins/cli/core/fx.dispatch-canvas-command.ts` confirms CLI tries local API server first when `--db` is not passed.
- So with the server running, CLI mutation executes inside the server process via RPC, not in a detached local-only repo.
- `packages/canvas-cmds/src/core/fx.canvas.ts` loads the canvas doc from `portal.automergeService.repo.find(...)`, waits for readiness, then mutation commands call `handle.change(...)` on that handle.
- This means the write path hits the server Automerge repo correctly.
- Frontend connects to Automerge websocket at `/automerge` via `apps/frontend/src/services/automerge.ts`.
- Added temporary browser logging to `packages/canvas/src/components/Canvas.tsx` and verified browser `DocHandle` change events do fire for CLI add.
- The logged payload showed the browser doc updated with the new ellipse, so CRDT sync transport is working.
- `packages/canvas/src/plugins/SceneHydrator/SceneHydrator.plugin.ts` only runs `loadCanvas()` from `context.hooks.initAsync`.
- No listener was found for `docHandle` change / update events that would re-run hydration or incrementally patch the scene.
- So refresh works because page reload rebuilds `CanvasService`, which re-runs one-time hydration from the latest doc snapshot.
- Conclusion: this is not a transport bug. It is missing live scene reaction support and now lives in A25.

## NOTES
Category: bug.

### Scope
This is about image paste into agent harnesses running inside the hosted terminal widget in canvas.
Examples:
- pi
- opencode
- claude
- codex

Text paste works.
Image paste works on localhost/dev and binary.
Image paste fails on deployed server via domain.

### Code path inspected
Frontend terminal paste handling lives in:
- `packages/canvas/src/components/terminal/GhosttyTerminalMount.tsx`

Key behavior now:
- text paste uses `event.clipboardData.getData(...)` and routes through `term.paste(text)`
- image paste uses either:
  - direct clipboard `File` payloads, or
  - fallback `navigator.clipboard.read()` image items
- image payload is uploaded to PTY temp storage and returned remote path is inserted into terminal as shell-escaped text
- unsupported non-text payloads still fall back to raw `"\x16"` (Ctrl+V)

Server HTTP path inspected:
- `apps/cli/src/plugins/server/http.ts`

Current finding:
- server does not set any clipboard-related `Permissions-Policy` headers
- server also does not add any extra response headers that would help diagnose clipboard gating

### Most likely root cause
Not a browser permission bug.

Manual repro on deployed app shows:
- `window.isSecureContext === true`
- `navigator.clipboard` exists
- terminal paste debug logs fire on deploy
- non-text clipboard is detected on deploy
- frontend handles it with `term.input("\x16")`

So browser-side canvas code is doing what it was built to do.

Real root cause is architectural:
1. Text paste works because the browser can read text from the local clipboard and send the literal text into the terminal stream.
2. Image paste path does **not** send image bytes through the terminal.
3. Instead, frontend emits raw Ctrl+V (`"\x16"`) as a generic hint to the terminal app.
4. That only works when the terminal app can read the same machine clipboard.
5. In local dev/binary, terminal app and user clipboard may effectively live on the same machine, so app-specific image-paste helpers can work.
6. In deployed mode, terminal process runs on the remote server, so it cannot read the user's local OS clipboard.
7. Result: deploy receives Ctrl+V, but remote `pi` / `opencode` / `claude` / `codex` process has no direct access to browser user's clipboard image.

So deploy failure is expected with the current generic `Ctrl+V` bridge.

### Why text still works
Text is browser-mediated.
The browser reads clipboard text and forwards the text payload into the terminal.

### Why image does not
Image is not browser-mediated today.
Frontend only forwards a control signal, not the actual file/blob bytes.

### Consequence
Fix will require an explicit browser-to-remote attachment path.
Generic terminal Ctrl+V is not enough for remote image paste.

### Important repo evidence
`GhosttyTerminalMount.tsx` does detect non-text clipboard payloads correctly on deploy.
Observed deploy logs:
- `[terminal:paste] paste event`
- `[terminal:paste] handled non-text paste { method: 'term.input', signal: '\\x16' }`

That proves:
- paste listener is attached
- focus path is good enough
- browser exposes non-text payload to the page
- frontend decision path is already taken

Failure happens after that point, in the remote terminal-app model, not in browser paste detection.

There is no repo evidence that terminal paste code clears clipboard contents.
Searched for clipboard writes/clears.
Found:
- copy helpers elsewhere in app
- no terminal-path clipboard write after paste
- no terminal-path clipboard clear after paste

So the flaky "clipboard got removed after paste" symptom is likely one of:
- browser or OS clipboard ownership behavior
- downstream terminal app behavior after receiving Ctrl+V
- false read caused by focus/permission race where second paste sees no readable payload

### Best next repro check
Open deployed app and run:
```js
localStorage.setItem('vibecanvas:terminal:debug', '1')
```
Then paste image once and inspect console for:
- `[terminal:paste] paste event`
- `[terminal:paste] fallback readText failed`
- `[terminal:paste] fallback read failed`
- reasons like permission denied, insecure context, or clipboard unavailable

Also inspect in console:
```js
window.isSecureContext
navigator.clipboard
```

Expected smoking gun:
- `window.isSecureContext === false`, or
- `navigator.clipboard.read()` rejected on deployed origin

### Chosen direction
Must stay terminal-level and harness-agnostic.

Chosen fix shape:
- intercept image paste in terminal widget
- upload clipboard image bytes to backend temp storage
- add new API in `packages/api-pty` for temp image upload
- API returns full remote absolute path
- terminal UI inserts that returned path as plain text into terminal input

Why this fits:
- no harness-specific integration
- works over remote deploy because payload crosses browser/server boundary explicitly
- keeps terminal abstraction: resulting UX is still text inserted into terminal

Limit:
- inserted path is only a file path, not a true attachment
- harness/user must still know what to do with that path
- but this is the best generic terminal-level behavior

Non-fix:
- more clipboard permissions will not solve the remote clipboard boundary by itself
- generic Ctrl+V alone will stay unreliable for remote image paste
- fake PTY clipboard/image event is not a solid generic path

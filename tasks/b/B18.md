# B18 - canvas delete unreadable-canvas errors are inconsistent and can print canvas name as `undefined`

## Plan
- Reproduce `canvas delete --json` against one readable-ish and one unreadable listed canvas using fake ids.
- Compare target-not-found vs document-unavailable behavior across canvases selected by name.
- Trace why one path reports `canvas 'undefined'` while other paths report document-open failure with `canvasId: null`.

## NOTES
- Category: bug.
- Status: open.
- Found during the divergent real-dev canvas CLI experiment with token `aad66061`.
- `delete --json` does not fail consistently when the selected canvas cannot be opened or cannot resolve the target ids.
- For `hello`, deleting a fake id returned `CANVAS_DELETE_TARGET_NOT_FOUND`, but the message incorrectly said `Target ids were not found in canvas 'undefined'` even though `canvasId` was present and `canvasNameQuery` was `hello`.
- For `test ground`, `recorder`, `power`, and `ok`, deleting the same fake id returned `CANVAS_DELETE_FAILED` with `Document automerge:... is unavailable` and `canvasId: null`.
- This looks like both an error-formatting bug and an inconsistent command-state bug: the delete path does not report selected canvas identity the same way across failure modes.

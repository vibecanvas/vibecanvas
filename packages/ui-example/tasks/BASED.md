# BASED

## TODO

- [ ] Separate runtime Konva group semantics from canvas semantic group semantics in `packages/canvas`.
  - Problem summary:
    - The codebase currently treats any runtime `Konva.Group` / `render.Group` as if it were a user-visible canvas group (`TGroup`).
    - That assumption used to mostly hold when runtime groups were only created by the actual group feature.
    - It no longer holds now that hosted components use a runtime `Konva.Group` as an implementation detail for one canvas element.
    - Result: hosted component host is misclassified as a canvas group by multiple systems.
  - User-visible bad behavior to preserve as repro/examples:
    - Selecting hosted component shows dashed/group-style transformer behavior meant for grouped content.
    - Context menu can offer `Ungroup` on hosted component, which is wrong.
    - Any feature branching on `instanceof render.Group` may incorrectly treat hosted component host as semantic `TGroup`.
    - Render-order persistence can classify hosted component as persisted `group` instead of persisted `element`.
  - Root cause in one sentence:
    - The code conflates two different layers:
      - render-engine primitives (`Konva.Group`, `Konva.Shape`), and
      - canvas document semantics (`TGroup` vs `TElement`).
  - Required conceptual split:
    - A runtime node can be one of:
      - scene node: selectable/renderable thing in Konva
      - semantic canvas group: corresponds to persisted `TGroup`
      - semantic canvas element host: corresponds to persisted `TElement`, even if implemented as `Konva.Group`
    - Hosted component host must be:
      - scene node: yes
      - semantic canvas group: no
      - semantic canvas element host: yes
  - Fix strategy:
    - Introduce explicit semantic predicates/helpers instead of raw `instanceof render.Group` checks.
    - Use semantic meaning for behavior, not runtime class.
    - Prefer checks like:
      - `editor.toGroup(node) !== null` for semantic canvas group
      - `editor.toElement(node) !== null` for semantic canvas element host
      - helper predicates if needed, e.g. `fxIsCanvasGroupNode`, `fxIsCanvasElementHostNode`, `fxIsSceneNode`
    - Real canvas groups should be explicitly marked or resolvable through existing group registries.
    - Hosted component runtime group must never be considered `TGroup`.
  - Highest-priority files to audit and fix first:
    - `packages/canvas/src/new-services/render-order/RenderOrderService.ts`
      - Current bug pattern: classifies `Konva.Group` as persisted `group` by runtime class alone.
      - Required change: persist as `group` only if semantic canvas group; otherwise persist as `element` if `editor.toElement(node)` succeeds.
      - Why this is P0: hosted component is a runtime group but document element.
    - `packages/canvas/src/new-plugins/transform/Transform.plugin.ts`
      - Current bug pattern: single selected `render.Group` gets group transformer behavior by raw class check.
      - Required change: `isSingleGroupSelection` and related logic must use semantic canvas-group detection, not `instanceof render.Group` alone.
      - Also verify serialization of selected nodes prefers semantic element host/group handling.
    - `packages/canvas/src/new-plugins/group/Group.plugin.ts`
      - Current bug pattern: selected `render.Group` can get group context actions such as `Ungroup`.
      - Required change: only semantic canvas groups qualify for group actions.
    - `packages/canvas/src/new-plugins/group/tx.ungroup-selection.ts`
      - Current bug pattern: reverse search for selected `render.Group` can pick hosted component host.
      - Required change: ungroup must operate only on semantic canvas groups.
  - Next-priority files to audit after P0:
    - `packages/canvas/src/new-plugins/select/fn.get-selection-path.ts`
      - Uses `render.Group` for semantic nesting path.
      - Hosted component runtime group should not imply semantic group nesting.
    - `packages/canvas/src/core/fn.filter-selection.ts`
      - Uses parent `render.Group` as subselection/group hierarchy signal.
      - Must distinguish semantic group hierarchy from runtime host internals.
    - `packages/canvas/src/new-plugins/context-menu/ContextMenu.plugin.ts`
      - Uses `render.Group` broadly in path and target resolution.
      - `targetGroup` should only be non-null for semantic canvas groups.
    - `packages/canvas/src/new-plugins/scene-hydrator/SceneHydrator.plugin.ts`
      - Builds maps of runtime groups and walks group containers.
      - Must avoid assuming every runtime group corresponds to persisted `TGroup`.
    - `packages/canvas/src/new-plugins/select/tx.delete-selection.ts`
      - Restore/delete logic treats runtime groups specially.
      - Must separate scene-parent/container behavior from semantic `TGroup` behavior.
  - Serialization/audit pattern to grep for and replace carefully:
    - `instanceof render.Group`
    - `instanceof Konva.Group`
    - `node instanceof render.Group || node instanceof render.Shape`
    - `parent instanceof render.Group ? parent.id() : null`
    - any place where runtime class directly determines whether node is a persisted `group`
  - Important expected semantics after fix:
    - Hosted component:
      - selectable: yes
      - transformable: yes
      - ungroupable: no
      - group context actions: no
      - persisted as `TElement` with `data.type = "custom"`
      - may use `Konva.Group` internally without becoming semantic `TGroup`
    - Real canvas group:
      - selectable: yes
      - group transformer semantics: yes
      - ungroupable: yes
      - persisted as `TGroup`
  - Suggested implementation approach:
    - Step 1: add one shared semantic helper in `core` or `new-plugins/group`.
      - Example responsibility:
        - semantic canvas group detection via `editor.toGroup(node)` or explicit attr.
        - semantic element-host detection via `editor.toElement(node)`.
    - Step 2: update P0 files to use semantic helper instead of raw runtime class checks.
    - Step 3: update selection/context-menu/scene-hydrator/delete flows.
    - Step 4: audit all `parentGroupId` serializers so only semantic canvas groups become `parentGroupId`.
  - Acceptance criteria:
    - Selecting hosted component does not trigger semantic group-only behavior.
    - Hosted component never shows `Ungroup` in context menu.
    - Render-order persistence treats hosted component as element, not group.
    - Real grouped content still supports group/ungroup correctly.
    - No feature relies on raw `Konva.Group` class alone where semantic `TGroup` meaning is intended.
